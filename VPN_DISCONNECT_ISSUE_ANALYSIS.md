# 🔍 Анализ проблемы с отключением VPN

**Дата:** 2026-01-22  
**Проблема:** Ошибка 500 при отключении, устройство остается в состоянии "подключено"

---

## 📊 Анализ логов

### Проблемная последовательность:

1. **Несоответствие состояния:**
   ```
   VpnService._syncConnectionState: Состояние на сервере: connected=true, локальное: _isConnected=false
   ```

2. **Попытка отключения:**
   ```
   VpnService._forceDisconnectOnServer: Начало отключения на сервере (device_id: UKQ1.230924.001)
   VpnService._forceDisconnectOnServer: ⚠️ Сервер вернул ошибку 500 при отключении (время: 915ms)
   ```

3. **Попытка подключения:**
   ```
   VpnService.connect: DioException поймана, statusCode: 400
   VpnService.connect: errorMessage: Устройство уже подключено
   ```

### Корневая причина:

**Устройство в БД имеет `is_active=True`, но реально не подключено.**

При попытке отключения:
- Происходит ошибка при удалении WireGuard пира (SSH ошибка, пир не найден, и т.д.)
- Ошибка приводит к rollback транзакции
- Состояние в БД **не очищается**
- При следующей попытке подключения сервер видит устройство как подключенное

---

## 🔧 Реализованные исправления

### 1. Улучшена обработка ошибок в `disconnect_device`

**Файлы:**
- `backend/services/connection_orchestrator.py`
- `backend/application/services/connection_orchestrator.py`

**Изменения:**
- ✅ **Всегда очищаем состояние в БД**, даже если удаление пира не удалось
- ✅ Добавлен параметр `force` для принудительного отключения
- ✅ Улучшена обработка исключений - не валим транзакцию при ошибках удаления пира
- ✅ Сохраняем `server_id` и `ip_address` до изменения device

**Ключевая логика:**
```python
# Удаляем пир (может не удаться - не критично)
try:
    wg_manager.remove_peer_from_server(device, server)
except Exception as e:
    logger.warning(f"Ошибка удаления пира: {e}. Продолжаем отключение в БД.")

# ВСЕГДА очищаем состояние в БД (критично!)
device.is_active = False
device.current_server_id = None
self.db.commit()  # Состояние очищено даже если удаление пира не удалось
```

### 2. Автоматический fallback на force disconnect

**Файлы:**
- `backend/services/vpn_operations_service.py`
- `backend/application/services/vpn_operations_service.py`

**Изменения:**
- ✅ При ошибке обычного отключения автоматически пробуется force disconnect
- ✅ Улучшена обработка ошибок - не возвращаем 500 если состояние очищено

### 3. Добавлен параметр `force` в API

**Файл:** `backend/api/vpn.py`

**Изменения:**
- ✅ Добавлен параметр `force` в `DisconnectVPNRequest`
- ✅ Поддержка принудительного отключения через API

### 4. Улучшена обработка на клиенте

**Файл:** `mobile-app/lib/services/vpn_service.dart`

**Изменения:**
- ✅ При ошибке 500 пробуем force disconnect автоматически
- ✅ Улучшена логика повторных попыток

## 🧪 Проверка исправления

### Тест 1: Обычное отключение

```bash
curl -X POST http://localhost:8000/api/vpn/disconnect \
  -H "Authorization: Bearer TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"device_id": "TEST_DEVICE"}'
```

**Ожидаемый результат:** 200 OK, состояние очищено

### Тест 2: Принудительное отключение

```bash
curl -X POST http://localhost:8000/api/vpn/disconnect \
  -H "Authorization: Bearer TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"device_id": "TEST_DEVICE", "force": true}'
```

**Ожидаемый результат:** 200 OK, состояние очищено (без попытки удаления пира)

### Тест 3: Проверка состояния после отключения

```sql
SELECT is_active, current_server_id FROM devices WHERE device_id = 'TEST_DEVICE';
-- Должно быть: is_active = false, current_server_id = NULL
```
