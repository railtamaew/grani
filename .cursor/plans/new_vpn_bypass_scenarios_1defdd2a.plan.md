# Новые сценарии обхода: план выполнения

Сценарии описаны в [docs/VPN_BYPASS_SCENARIOS.md](../../docs/VPN_BYPASS_SCENARIOS.md). Стратегия выполнения: [docs/VPN_BYPASS_EXECUTION_PLAN.md](../../docs/VPN_BYPASS_EXECUTION_PLAN.md).

## Исходные условия

- **Сервер:** один (Венгрия), XRay + WireGuard. Используем только когда нужно проверить реальный трафик в реальной сети.
- **Цель:** разработать и довести сценарии в лаборатории до состояния «работает, трафик идёт», минимизируя ручные проверки; тест в мобильном приложении — только на конечном этапе.

---

## Три фазы

| Фаза | Где | Что делаем | Ручной тест |
|------|-----|------------|-------------|
| **1. Лаборатория** | Локальная среда, скрипты, симулятор DPI, при необходимости один тестовый хост (не обязательно Венгрия) | Реализация PoC, проверка обхода блокировок, сбор/анализ трафика (pcap, JA3 и т.д.) | Минимум: только ключевые проверки «подключилось / трафик пошёл» вручную в конце фазы по сценарию |
| **2. Интеграция** | Бэкенд + при необходимости сервер (Венгрия) | Подключение к API, выдача конфигов, один реальный endpoint для проверки трафика | Ручная проверка только когда нужно убедиться в реальной сети |
| **3. Приложение** | Мобильное приложение | Выбор протокола в UI, полный цикл подключения, E2E | Ручное тестирование на устройстве/эмуляторе — только на этом этапе |

До конца фазы 1 сервер Венгрия и приложение не обязательны: всё доводится до «идеала» в лаборатории.

---

## Фаза 1: Лаборатория

### 1.1 Инфраструктура лаборатории (один раз)

- **Симулятор DPI/SNI:** прокси (mitmproxy или свой), правила: блок по SNI, по отпечатку (JA3/JA4), RST по признакам.
- **Скрипты анализа трафика:** запись pcap, извлечение JA3/JA4, размеры первых N пакетов, сравнение с эталоном (переиспользовать `backend/scripts/bypass_lab_capture_traffic.py`, `backend/scripts/vpn_tester.py`).
- **Тестовый клиент:** минимальный скрипт или утилита (не приложение), подключающаяся к PoC-серверу по выбранному протоколу и гоняющая тестовый трафик.

Оценка: 1–2 недели на стенд; дальше переиспользование для всех сценариев.

### 1.2 Порядок проработки сценариев в лаборатории

1. **Сценарий 6 (Bootstrap)** — PoC: получение конфига через DoH или HTTPS к «прикрывающему» домену. Тесты: доступность DoH, загрузка конфига. Ручной тест: один раз проверить, что конфиг скачивается и парсится.
2. **Сценарий 4 (Boring HTTPS)** — PoC: клиент (скрипт) + сервер (HTTP endpoint): POST с payload, ответ, туннель в теле HTTPS. Тесты: pcap, симулятор DPI не блокирует. Ручной тест: «клиент подключается, запрос через туннель — видим ответ» (curl/скрипт).
3. **Сценарий 1 (Traffic Mimicry)** — PoC: Boring HTTPS + подстройка TLS-отпечатка и размеров/таймингов под эталон. Тесты: сравнение pcap с эталоном; симулятор не режет. Ручной тест: один прогон «подключение + запрос через туннель».
4. **Сценарий 3 (Adaptive Strategy)** — PoC: логика выбора стратегии по зондам; переключение между режимами. Тесты: unit/скрипты с разными правилами блокировки. Ручной тест: 1–2 сценария «включили блокировку X → клиент переключился на Y и трафик пошёл».
5. **Сценарии 2, 5, 7** — по приоритету: PoC в lab, те же принципы (автоматические проверки → минимальная ручная проверка «трафик пошёл»).

Итог фазы 1: для каждого выбранного сценария — работающий PoC и доказательство в лаборатории; ручные проверки точечные.

---

## Фаза 2: Интеграция

Включается только после успешной лаборатории по сценарию.

- **Бэкенд:** отдельный менеджер протокола, выдача конфига через API; при необходимости feature flag.
- **Сервер (Венгрия):** подключаем только когда нужно проверить «в реальной сети»: поднять endpoint на этом сервере, выдать конфиг с его URL, один раз вручную проверить подключение и трафик. До этого можно тестировать на локальном или отдельном тестовом хосте.
- **Ручной тест:** минимально — «бэкенд отдаёт конфиг → тестовый клиент подключается к реальному endpoint и трафик идёт». Приложение на этом этапе можно не трогать.

---

## Фаза 3: Приложение (только конечный этап)

- Подключение нового протокола в приложении: парсинг конфига, выбор в UI, адаптер (например BoringHttpsAdapter), установка/разрыв.
- Тестирование: вручную на устройстве/эмуляторе — полный цикл (сервер → протокол → подключение → проверка трафика). При желании — E2E позже.
- Правило: до этой фазы все задумки доводятся до идеала в лаборатории и на интеграции; в приложении только финальная проверка и полировка UX.

---

## Схема по этапам

```
Лаборатория (lab)
  → PoC по сценариям 6 → 4 → 1 → 3 → (2, 5, 7)
  → симулятор DPI + pcap + скрипты
  → минимальный ручной тест: «подключилось, трафик пошёл»

Интеграция
  → бэкенд: менеджеры, API, конфиги
  → при необходимости: один раз поднять endpoint на сервере (Венгрия) и проверить трафик вручную
  → без приложения или с минимальным клиентом

Приложение
  → только на конечном этапе: выбор протокола, полный цикл, ручное тестирование в приложении
```
