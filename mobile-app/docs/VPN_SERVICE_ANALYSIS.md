## Оценка протоколов

### Приоритет: Reality > VLESS > VMESS

- **xray_reality** (443): первый выбор — маскировка под TLS, устойчивость к DPI.
- **xray_vless** (4443): резерв при отсутствии Reality.
- **xray_vmess** (8443): последний в цепочке fallback.

### Общая архитектура
- Xray-core (libXray) в отдельном процессе `:xray`.
- tun2socks в отдельном процессе `:tun2socks` — каждый connect получает новый процесс, что устраняет `tcp_bind_to_netif failed` при reconnect.
- TUN → tun2socks → Xray SOCKS-in (127.0.0.1:10808) → uplink.

### Замечания
1. **Reality** — наилучший баланс скорости и обхода блокировок.
2. **VLESS** — меньше overhead, чем VMESS.
3. **VMESS** — устаревший, но поддерживается для совместимости.
4. Все протоколы используют один и тот же путь: tun2socks + Xray SOCKS-in.

## Общая оценка сервиса

### Сильные стороны
- Раздельные процессы для Xray и tun2socks — стабильный reconnect.
- Кэширование конфигурации — быстрый reconnect из кэша.
- Fallback URL API при блокировке оператора.
- Health-probe для Reality (интервал 30 с).
- MTU подбор под Wi‑Fi (1500) и мобильную сеть (1280).

### Потенциальные улучшения
1. **ConnectionLogger** — при VPN включён отправка логов идёт через туннель; при таймаутах (10 с) возможны задержки. Рассмотреть отправку логов в фоне без блокировки UI.
2. **Краш tun2socks** — при `stopTun2Socks` возможен `pthread_mutex_lock on destroyed mutex` в BadVPN. Процесс :tun2socks убивается в `onDestroy()`, поэтому краш не ломает основной процесс, но логи остаются.
3. **API disconnect** — запрос `/vpn/disconnect` с таймаутом 8 с выполняется параллельно с локальным disconnect; при медленной сети отключение UI уже произойдёт, задержки не вносит.

### Рекомендации
- Продолжать мониторить reconnect (кэш, первое подключение).
- Сокращённые паузы протестировать на разных устройствах (в т.ч. слабые).
- При повторении крашей BadVPN рассмотреть переход на другую реализацию tun2socks или обёртку с игнорированием SIGABRT в :tun2socks перед kill.
