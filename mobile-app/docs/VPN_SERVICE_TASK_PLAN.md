## Фаза 3 — Архитектура протоколов (крупный рефакторинг)

Вынесено за рамки предыдущего плана; выполнять отдельно, после стабилизации фаз 1–2.

| # | Задача | Описание |
|---|--------|----------|
| 3.1 | Интерфейс VpnProtocolHandler | Определить общий контракт: получение конфига, применение конфига, отключение, при необходимости проверка доступности. Сигнатуры не должны зависеть от конкретного протокола (Xray, WireGuard, Singbox и т.д.). |
| 3.2 | Реализации по протоколам | Вынести логику в отдельные классы: XrayProtocolHandler (текущая _connectXray и связанное), WireGuard, Singbox и т.д. Каждый реализует общий интерфейс и инкапсулирует вызовы нативного слоя и парсинг конфигов. |
| 3.3 | Интеграция в VpnService | В connect() выбирать handler по _selectedProtocol и вызывать его методы вместо прямых вызовов _connectXray и т.п. Сохранить текущее поведение (этапы, логирование, lastConnectionErrorMessage, таймауты). |
| 3.4 | Тесты протоколов | Тесты для каждого handler’а с моками (API, нативный слой); тесты VpnService на переключение протоколов и вызов нужного handler’а. |

**Критерий готовности фазы 3:** Подключение по каждому протоколу идёт через свой handler; VpnService не содержит протокол-специфичной логики внутри; тесты покрывают контракт и основные сценарии.

**✅ Фаза 3 выполнена:** 3.1 — интерфейс `VpnProtocolHandler` и `ProtocolConnectParams`. 3.2 — делегаты для всех протоколов: `_XrayHandlerDelegate`, `_WireGuardHandlerDelegate`, `_BoringHttpsHandlerDelegate`, `_OpenVPNCloakHandlerDelegate`; `getHandlerFor(protocol)` возвращает handler для каждого протокола; встроенный switch в `_connectInternal` удалён, подключение только через handler. 3.3 — интеграция (handlerFactory + getHandlerFor). 3.4 — тесты контракта и тест «getHandlerFor возвращает handler для всех протоколов» в `vpn_service_mocks_test.dart`.
