# Скорость подключения Xray

## Почему повторное подключение было таким же долгим, как первое

При каждом подключении (включая повторное) приложение вызывало **только** `POST /vpn/xray/create-client`. Этот эндпоинт на бэкенде каждый раз:

1. Читает конфиг Xray с VPN-сервера по SSH
2. Добавляет/обновляет клиента в конфиге
3. Применяет конфиг на сервер (запись файла + `systemctl reload xray`)

Из-за этого один запрос занимает **15–25 секунд** (сетевые задержки + несколько SSH-операций). Кэш конфига на бэкенде ускоряет только чтение при повторных запросах к одному серверу, но сам вызов create-client всё равно делался при каждом подключении, поэтому повторное подключение оставалось таким же долгим.

## Что сделано для ускорения

### 1. Таймаут: 30 с только при первом create-client

- **Первое** подключение к серверу (нет `client_id` для этого сервера): таймаут create-client **30 секунд**, чтобы бэкенд успел выполнить SSH (чтение конфига, применение, reload).
- **Повторная** попытка (fallback после неудачного GET или после ошибки «уже подключено»): таймаут **15 секунд**.
- При успешном GET config повторное подключение обходится без create-client, таймаут не используется.

### 2. Быстрый путь при повторном подключении (мобильное приложение)

Если у приложения уже есть `client_id` для **текущего** сервера (тот же сервер, что и при последнем подключении), оно **сначала** запрашивает конфиг через:

- **GET** `/vpn/xray/config/{client_id}`

Этот эндпоинт не выполняет SSH: конфиг собирается из БД (устройство, сервер, протокол). Ответ приходит за **0.5–2 секунды**.

- При успешном ответе (200) приложение использует полученный конфиг и **не вызывает** create-client.
- При ошибке (404, таймаут 5 с) или при первом подключении (нет `client_id` для этого сервера) выполняется обычный вызов create-client.

В результате **повторное подключение к тому же серверу** занимает 1–2 секунды вместо 15+.

### 3. Рекомендации по бэкенду (первое подключение)

Чтобы ускорить именно **первое** подключение к серверу:

- **Celery**: убедиться, что воркер Celery запущен и задача `vpn.apply_xray_config` выполняется. Тогда create-client возвращает ответ сразу после обновления конфига в памяти и постановки задачи в очередь; применение конфига на VPN-сервер идёт в фоне.
- **Кэш конфига**: бэкенд уже кэширует конфиг Xray по серверу (TTL 10 мин). При повторных create-client к одному серверу чтение конфига с VPN идёт из кэша, без SSH.

## Итог

| Сценарий | Раньше | Сейчас |
|----------|--------|--------|
| Первое подключение к серверу | 15–25 с, таймаут 15 с (риск обрыва) | 15–30 с, таймаут **30 с** (запас по времени) |
| Повторное подключение к тому же серверу | 15–25 с | **1–2 с** (GET config вместо create-client) |
| Повтор create-client (после «уже подключено» или fallback) | 15 с | 15 с |

## Другие проблемы и возможные оптимизации

### Бэкенд

1. **Первое подключение всё ещё 15–25 с**  
   Причина: при промахе кэша бэкенд один раз читает конфиг по SSH (~5–15 с), затем ставит задачу Celery на применение конфига. Если Celery не запущен, применение делается синхронно (ещё один SSH).  
   **Оптимизации:**  
   - Убедиться, что воркер Celery запущен и обрабатывает `vpn.apply_xray_config` — тогда create-client отвечает сразу после обновления конфига в памяти.  
   - Опционально: «прогрев» кэша при старте API или при первом запросе к серверу (фоновое чтение конфига по SSH), чтобы следующий create-client шёл уже по кэшу.

2. **Кэш конфига Xray — in-memory**  
   При перезапуске бэкенда кэш пустой, первое подключение к каждому серверу снова медленное.  
   **Оптимизация:** при необходимости — вынести кэш в Redis с TTL (требует настройки инфраструктуры).

3. **Быстрый путь «клиент уже есть»**  
   Сейчас create-client всегда обновляет конфиг и применяет его на сервер. Если у устройства уже есть `vpn_client_id` и этот клиент есть в текущем (кэшированном) конфиге сервера, можно сразу вернуть конфиг без записи на VPN (аналогично GET /config, но через один эндпоинт). Требует доработки API.

4. **Один SSH-сеанс при отключённом Celery**  
   При синхронном применении конфига сейчас: один SSH на чтение, затем при apply — ещё один на запись и reload. Можно объединить «чтение → изменение → запись → reload» в один SSH-сеанс, чтобы сократить число подключений и задержки.

### Мобильное приложение

5. **Проверка подключения после коннекта**  
   `_verifyConnection` вызывает `_getCurrentIpAddress` (api.ipify.org) и `_testDnsResolution` (google.com) с таймаутами по 15 с. Выполняются последовательно.  
   **Оптимизация:** запускать проверку IP и DNS параллельно и/или уменьшить таймауты (например, 8–10 с).

6. **Жёсткая пауза 1 с после старта Xray (Android)**  
   В `XrayNativeWrapperTun2Socks` после `startVpn` вызывается `Thread.sleep(1000)`.  
   **Оптимизация:** заменить на ожидание по флагу «туннель готов» или сократить до 300–500 мс с проверкой готовности.

7. **Повтор при таймауте GET config**  
   При таймауте GET /config (5 с) приложение сразу идёт в create-client. Можно один раз повторить GET с коротким таймаутом перед fallback на create-client.

### Общее

8. **Список серверов и здоровье**  
   Загрузка списка серверов и их `health_status` уже кэшируется на бэкенде (5 мин). При необходимости можно добавить кэш на клиенте на 1–2 мин, чтобы не дергать API при частых переключениях экранов.
