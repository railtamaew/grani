# План упрощения и ускорения подключения (сессии/девайсы/Xray)

## Контекст

- Текущее наблюдение: подключение по Wi-Fi может занимать до ~1 минуты.
- На практике задержка складывается из control-plane (API/сессии/девайсы/apply-state) и data-plane (старт Xray/tun2socks/probes).
- Дополнительно на стабильность и время влияет внешний шум на портах (сканеры), что уже частично снижено (iptables + fail2ban).

## Целевое состояние

- Подключение (Wi-Fi) p50 <= 8s, p95 <= 15s.
- Повторное подключение к тому же серверу p95 <= 5s.
- Отсутствие "ложного connected" (когда локально connected, но интернет не идет).
- Один понятный маршрут подключения без лишних fallback-циклов.

## Что влияет на долгий коннект

1. `create-client` через SSH (чтение/изменение/apply конфига сервера).
2. Ожидание `apply-state` и ACK-пайплайна.
3. Повторная валидация/регистрация девайса в неудачных ветках.
4. Последовательные post-connect проверки (IP/DNS/API) с длинными таймаутами.
5. Дополнительные паузы в нативном старте туннеля.
6. Протокольные переключения в рамках одной попытки.
7. Шум/скан портов (увеличивает конкуренцию за ресурсы, мусорит логи).

## Упрощение по слоям

### 1) Session/Device слой (основной источник сложности)

Цель: сделать один "прямой" путь и убрать лишние ветки.

- Ввести единый runtime-контракт:
  - один активный `connection_session_id` на попытку;
  - все события стадии жестко привязывать к этой сессии;
  - старые/параллельные попытки сразу гасить (`stale session`).
- Упростить device lifecycle:
  - загрузка `device_id` строго до network-вызовов;
  - `ensureDeviceRegistered` запускать только при отсутствии валидной device-связки;
  - кешировать "device verified" на короткое окно (например 5-10 минут), чтобы не дублировать проверки на каждый connect.
- Убрать циклические переходы:
  - запретить повторные `session/prepare -> create-client` в одной попытке более 1 раза.

### 2) Control-plane слой (API/apply-state)

Цель: минимизировать блокирующие шаги.

- Основной путь:
  - сначала `GET /vpn/xray/config/{client_id}` (быстрый путь без SSH);
  - `create-client` только при 404/отсутствии клиента.
- `create-client`:
  - при активном Celery возвращать ранний ответ (queue accepted), без ожидания полного SSH цикла;
  - для первого подключения сохранить длинный таймаут, но убрать двойные повторные вызовы в одном сценарии.
- `apply-state`:
  - считать `applied` только если:
    - `xray-v2` active,
    - нужные inbound-порты listening.

### 3) Data-plane слой (туннель на устройстве)

Цель: быстрее и предсказуемее поднимать туннель.

- Сохранить full-stop старого ядра перед новым стартом (уже улучшено).
- Убрать жесткие sleep там, где можно ждать флаг готовности.
- Post-connect probes:
  - запускать параллельно;
  - сократить таймауты;
  - делать "мягкий degraded", не ломая стабильную сессию раньше времени.

### 4) Протоколы и fallback

Цель: не раздувать ветвление.

- Основной протокол: `xray_vless`.
- Fallback: `xray_reality` (одна попытка).
- Исключить протокольные "качели" в рамках одной connect-сессии.

## План работ (итерации)

### Итерация A (1-2 дня): инструментирование времени

- Добавить тайм-метки этапов:
  - `connect_start`
  - `device_ready`
  - `config_ready`
  - `apply_state_ready`
  - `native_connected`
  - `internet_ready`
- Собирать p50/p95 по этапам.

### Итерация B (2-3 дня): упрощение session/device

- Единая state-machine для connect.
- Жесткое ограничение ретраев в одной сессии.
- Кеш "device verified window".

### Итерация C (2-3 дня): control-plane и applied guard

- Закрыть двойные create-client ветки.
- Проверка `xray-v2 active + listening ports` перед финальным applied.
- Проверка деградаций и отказов по новым метрикам.

### Итерация D (1-2 дня): data-plane ускорение

- Параллельные probes.
- Уменьшение блокирующих пауз.
- Финальная калибровка таймаутов.

## KPI и критерии приемки

- Wi-Fi connect:
  - p50 <= 8s
  - p95 <= 15s
- Повторный connect к тому же серверу: p95 <= 5s.
- Доля сессий с `connected_degraded_retry` в первые 20 секунд снизить минимум на 50%.
- Отсутствие дублей активных connect-сессий для одного device.

## Риски

- Слишком агрессивное сокращение таймаутов может дать ложные fail на медленных сетях.
- Fallback-путь нужно оставить, иначе в части сетей ухудшится доступность.
- Изменения state-machine требуют строгой телеметрии, иначе сложно диагностировать регрессии.

## Что проверять после каждого релиза

- Логи `client_logs` по стадиям сессий.
- Частоту `connection_error` по кодам.
- Разрез по `network_type=wifi/mobile`.
- Время этапов `config_ready -> native_connected -> internet_ready`.
