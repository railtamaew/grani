# План выполнения: лаборатория → интеграция → приложение

Согласованная стратегия разработки и тестирования сценариев из [VPN_BYPASS_SCENARIOS.md](VPN_BYPASS_SCENARIOS.md).

## Исходные условия

- **Сервер:** один (Венгрия), XRay + WireGuard. Используем только когда нужно проверить реальный трафик в реальной сети.
- **Цель:** разработать и довести сценарии в лаборатории до состояния «работает, трафик идёт», минимизируя ручные проверки; тест в мобильном приложении — только на конечном этапе.

## Три фазы

**Фаза 1. Лаборатория** — локальная среда, симулятор DPI/SNI, скрипты анализа pcap, тестовый клиент (не приложение). PoC по сценариям 6 → 4 → 1 → 3 → (2, 5, 7). Минимальный ручной тест: «подключилось, трафик пошёл». Сервер Венгрия и приложение не обязательны.

**Фаза 2. Интеграция** — бэкенд (менеджеры, API, конфиги); при необходимости один раз поднять endpoint на сервере (Венгрия) и проверить трафик вручную. Без приложения или с минимальным клиентом.

**Фаза 3. Приложение** — только на конечном этапе: выбор протокола в UI, полный цикл подключения, ручное тестирование на устройстве/эмуляторе.

## Порядок сценариев в лаборатории

1. Сценарий 6 (Bootstrap) — PoC DoH/прикрывающие домены; ручной тест: конфиг скачивается и парсится.
2. Сценарий 4 (Boring HTTPS) — PoC клиент + HTTP endpoint; pcap, симулятор; ручной тест: запрос через туннель, видим ответ.
3. Сценарий 1 (Traffic Mimicry) — PoC + TLS-отпечаток под эталон; сравнение pcap; ручной тест: один прогон.
4. Сценарий 3 (Adaptive Strategy) — логика выбора по зондам; ручной тест: блокировка X → переключение на Y.
5. Сценарии 2, 5, 7 — по приоритету, те же принципы.

## Инфраструктура лаборатории (один раз)

- Симулятор DPI/SNI (прокси с правилами: блок по SNI, JA3/JA4, RST).
- Скрипты: `backend/scripts/bypass_lab_capture_traffic.py`, `backend/scripts/vpn_tester.py`.
- Тестовый клиент: скрипт или утилита, не приложение.

Оценка: 1–2 недели на стенд.

## Чек-лист интеграции (фаза 2)

- Бэкенд: BoringHttpsManager, ConnectionOrchestrator._connect_boring_https, API supported_protocols при BORING_HTTPS_GATEWAY_URL (см. backend/core/config.py, backend/api/vpn.py).
- Gateway: POST /api/vpn/tunnel в backend/api/boring_https_gateway.py; при необходимости развернуть на сервере (Венгрия) и задать boring_https_gateway_url в настройках сервера или env.
- Ручная проверка: задать BORING_HTTPS_GATEWAY_URL, запустить бэкенд, выполнить bypass_lab_tunnel_client.py --url <gateway_url>; убедиться, что health и фрейм возвращают 200.

## Чек-лист приложения (фаза 3)

- В приложении: протокол boring_https в селекторе (protocol_selector_bottom_sheet.dart), VpnProtocol.boringHttps, BoringHttpsAdapter в Android, vpn_service.dart и модели server.dart (supportedProtocols). Ручной тест: выбор сервера → выбор протокола Boring HTTPS → подключение → проверка трафика на устройстве/эмуляторе.
