# План исправлений: удобство и скорость VPN

Цель: сократить время подключения (tap → «Подключено») с ~39 с до 5–15 с и улучшить восприятие сервиса пользователем.

---

## 1. Сокращение времени этапа «Проверка подключения» (приоритет: высокий)

**Проблема:** В логах `verify_connection` занимает ~19 с; IP и DNS проверки используют таймаут 15 с каждая и часто срабатывают по таймауту.

**Файлы:** [mobile-app/lib/services/vpn_service.dart](mobile-app/lib/services/vpn_service.dart)

**Действия:**

1. **Уменьшить таймауты запросов в проверках:**
   - `_getCurrentIpAddress()`: таймаут с 15 с до **4 с** (строка ~943: `Duration(seconds: 15)` → `Duration(seconds: 4)`).
   - `_testDnsResolution()`: таймаут с 15 с до **4 с** (строка ~958).

2. **Выполнять IP и DNS проверки параллельно**, а не последовательно:
   - В `_verifyConnection()` запускать `_getCurrentIpAddress()` и `_testDnsResolution()` через `Future.wait([...])` с общим ограничением по времени (например, `Future.wait([...]).timeout(Duration(seconds: 5))`).
   - Критерий успеха подключения оставить: трафик обнаружен И (IP проверка не критична для перехода в «Подключено»; при провале — только предупреждение).

3. **Ранний выход при наличии трафика:**
   - Сначала выполнять быструю проверку трафика (уже есть задержка 400 ms + при необходимости 800 ms повтор). Если `rx_bytes + tx_bytes > 0` — считать этап проверки успешным и не ждать завершения IP/DNS (или ждать их в фоне с коротким таймаутом).
   - Либо: считать подключение успешным по факту трафика; IP/DNS проверки запускать после перехода в состояние `connected` и только логировать/показывать предупреждение в UI при неудаче.

**Ожидаемый эффект:** этап verify_connection сократится с ~19 с до ~5–8 с.

---

## 2. Показ «Подключено» до завершения всех проверок (приоритет: высокий)

**Проблема:** Пользователь видит «Проверка подключения» до 20+ секунд, хотя туннель уже поднят и трафик идёт.

**Действия:**

1. В `connect()` после успешного `_applyXrayConfig()` / установки туннеля:
   - Сразу перейти в этап «Проверка подключения» с **ограничением по времени** (например, макс. 6 с на всю верификацию).
   - Критерий успеха: за отведённое время обнаружен трафик (`getTrafficStats()` > 0). Остальное (IP, DNS) — по возможности параллельно и без блокировки перехода в «Подключено».

2. Либо двухфазный сценарий:
   - Фаза 1: трафик есть → переход в `VpnConnectionState.connected`, обновление UI «Подключено».
   - Фаза 2: в фоне выполнить IP/DNS проверки; при неудаче — обновить подпись/бейдж (например, «Подключено, возможны ограничения») или показать одно предупреждение в UI без отключения VPN.

**Ожидаемый эффект:** Пользователь видит «Подключено» через 5–12 с после нажатия вместо ~39 с.

---

## 3. Оптимизация этапа apply_protocol / engine_init (приоритет: средний)

**Проблема:** В логах `apply_protocol` + `engine_init` ~18.5 с (ожидание запуска нативного XRay + tun2socks + первая проверка).

**Файлы:** нативная часть [VpnService.kt](mobile-app/android/app/src/main/kotlin/com/granivpn/mobile/VpnService.kt), [XrayNativeWrapper](mobile-app/android/app/src/main/kotlin/com/granivpn/mobile/XrayNativeWrapper.kt), Flutter [vpn_service.dart](mobile-app/lib/services/vpn_service.dart).

**Действия:**

1. Проверить, можно ли уменьшить фиксированную задержку перед проверкой «XRay запущен» (например, в `XrayNativeWrapper` или в `processXrayPackets` — сейчас 2 с ожидания после старта). Сократить до 1 с или до первой успешной проверки `isXrayRunning()` с коротким polling.

2. На стороне Flutter: не делать лишних ожиданий между вызовом `NativeVpnService.connect()` и стартом проверки трафика; запускать опрос трафика сразу после возврата из `connect()` (если API это позволяет).

3. Рассмотреть кэширование конфига/сервера при повторном подключении (уже частично есть), чтобы минимизировать время до первого байта.

**Ожидаемый эффект:** Сокращение engine_init на 1–3 с.

---

## 4. Диагностика и логирование (приоритет: низкий)

**Действия:**

1. При ответе 400 в `_syncConnectionState` логировать тело ответа (или код ошибки) для поддержки, без изменения поведения «игнорируем».

2. Сохранять в логи этапы с таймингами (уже есть `_lastConnectionTimingMs`); при желании — отправлять агрегированную метрику `tap_to_connected_ms` на бэкенд для аналитики.

---

## 5. Улучшение сообщений пользователю (приоритет: низкий)

**Действия:**

1. При таймауте проверки IP или DNS не показывать формулировки вроде «DNS не работает», если трафик уже есть и VPN фактически работает. Варианты: «Проверка DNS заняла больше времени» или не показывать предупреждение, а только логировать.

2. Текст прогресса «Проверка подключения» можно уточнить: например, «Проверка трафика…» на первом шаге и не держать этот текст 15+ секунд.
