# Анализ перегрузки сервера API

**Контекст:** пользователи сообщают о таймаутах при авторизации (bootstrap не получен, send-code не отвечает). С интернетом у клиентов всё в порядке — проверяем, нет ли перегрузки на нашей стороне.

---

## 1. Текущая конфигурация

| Компонент | Настройка |
|-----------|-----------|
| **Запуск API (Docker)** | `gunicorn main:app -w 4 -k uvicorn.workers.UvicornWorker` (4 воркера) |
| **Запуск API (main.py)** | `uvicorn.run(..., reload=False)` — без `workers`, один процесс при ручном запуске |
| **БД пул** | `pool_size=20`, `max_overflow=10` на процесс (в сумме до 30 соединений на воркер) |
| **Nginx** | `proxy_*_timeout 60s`, `worker_connections 1024` |
| **Rate limit (send-code)** | 5 запросов/мин с IP, 3 запроса/мин на email |

Классической «перегрузки» (нехватки воркеров при малой нагрузке) при 4 воркерах в Docker не должно быть. Риск — **блокировка воркера** одним долгим запросом.

---

## 2. Узкое место: синхронная отправка email

- **Эндпоинт** `POST /api/auth/send-code` вызывает `send_verification_email()` из `services.email_service`, которая синхронно дергает AWS SES / Yandex Postbox (`boto3.client(...).send_email()`).
- Вызов **синхронный** и выполнялся **в том же потоке**, что и event loop FastAPI/uvicorn. Пока один запрос ждёт ответа от SES (1–5+ секунд), **весь воркер блокируется** — в том числе не обрабатываются запросы к `/api/vpn/bootstrap` и другим эндпоинтам на этом воркере.
- При нескольких одновременных запросах send-code или при медленном ответе Postbox очередь на воркере растёт → клиенты получают таймауты (bootstrap, send-code).

**Исправление:** отправка email перенесена в потоковый пул (`core.async_utils.run_sync`), чтобы не блокировать event loop. Тогда один долгий вызов SES не блокирует обработку bootstrap и остальных запросов на этом воркере.

---

## 3. Рекомендации

1. **Деплой:** убедиться, что в проде API действительно запускается через gunicorn с несколькими воркерами (как в Dockerfile), а не одним процессом `uvicorn main:app`.
2. **Мониторинг:** при продолжении таймаутов смотреть логи API и Nginx в момент инцидента (латентность, 5xx, таймауты upstream), а также метрики БД и Redis при необходимости.
3. **Опционально:** для продакшена с несколькими воркерами использовать `rate_limit_store=redis` и `cache_backend=redis`, чтобы лимиты и кэш были общими для всех воркеров (см. `core/config.py`).

---

## 4. Внесённые изменения

- **`api/auth.py`**: вызовы `send_verification_email` в эндпоинтах `send_code` и `test_email_sending` выполняются через `await run_sync(send_verification_email, ...)`, чтобы не блокировать event loop.

После деплоя этих изменений воркеры не должны «залипать» на время отправки email, и запросы к bootstrap и send-code должны обрабатываться параллельно без лишних таймаутов из‑за блокировки.
