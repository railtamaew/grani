# Отчет об инциденте: Xray Reality и WireGuard (2026-01-23)

## Контекст
Пользователь инициировал подключение к двум протоколам (Xray Reality и WireGuard) к серверу `HU-BUD-01`. Подключение не удалось. Ниже — объединенный анализ по логам приложения, БД, API и сервера.

## Сводка (критические выводы)
1) **Xray Reality не запускается на клиенте**: `libXray` возвращает ошибку `empty "privateKey"` при разборе JSON-конфига. Это означает, что нативный Xray требует `privateKey` в `realitySettings` (или получает конфиг в несовместимом формате) и завершает запуск до сетевого подключения.
2) **WireGuard не стартует на устройстве**: на клиенте `GoBackend` падает по `TimeoutException`, хотя API считает подключение успешным и фиксирует подключение в БД.
3) **Состояния “подключено” и “реально работает” расходятся**: API и БД фиксируют подключение, но клиент не поднимает туннель. Это приводит к ошибочным `connected=true` на серверной стороне и принудительным disconnect в приложении.
4) **SSH/SFTP с сервером нестабильны** во время операций: `paramiko.transport - Dropping user packet because connection is dead`, что может приводить к неполному/частичному обновлению конфигов на сервере.

## Корневые причины
1) **Некорректный Reality‑конфиг для libXray**: отсутствие `privateKey` приводит к отказу запуска Xray‑core внутри клиента. Это блокер для Reality.
2) **Несогласованность состояния клиента и сервера**: сервер отмечает подключение успешным, но клиент не поднимает туннель.
3) **Нестабильные SSH/SFTP операции** при обновлении конфигов на сервере.
4) **WireGuard‑туннель не поднимается на клиенте** (Timeout) при формально успешном серверном connect.

## Риски для сервиса
- Ложные статусы подключения в UI/Backend → пользователь видит “подключено”, но трафик не идет.
- Повышенная нагрузка из‑за повторных попыток подключения/отключения.
- Потенциальная рассинхронизация конфигов Xray/WireGuard при обрывах SFTP.

## Рекомендации (приоритет)
1) **Reality конфиг для libXray**: подтвердить требуемый формат для используемого AAR. Если библиотека действительно требует `privateKey`, нужно обновить libXray/Xray‑core или перейти на схему, где privateKey не требуется на клиенте.
2) **Сделать верификацию “туннель поднят” перед отметкой `is_active=True`** (подтверждение со стороны клиента или health‑check).
3) **Стабилизировать SSH/SFTP**: повторный ретрай/валидация целостности файла после записи.
4) **WireGuard Timeout**: проверить причинy таймаута на устройстве (исключить конфликты с предыдущими сессиями/запущенным VPN, проверить MTU/permissions).
5) **Очистка дублей устройств** (уточнить, почему `UKQ1.230924.001` встречается у разных пользователей).
