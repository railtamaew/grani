# Тестирование и интеграция новых сценариев обхода блокировок

Документ описывает варианты тестирования и интеграции сценариев из [VPN_BYPASS_SCENARIOS.md](VPN_BYPASS_SCENARIOS.md) в сервис GRANI (backend, mobile-app, админ-панель). Использовать **перед выбором** конкретного сценария для реализации и в процессе внедрения.

Пошаговая лабораторная оценка (захват трафика, сравнение JA3/SNI): [VPN_BYPASS_LAB_EVALUATION.md](VPN_BYPASS_LAB_EVALUATION.md). Скрипт захвата: `backend/scripts/bypass_lab_capture_traffic.py`.

---

## 1. Тестирование до реализации (оценка сценария)

Цель: оценить жизнеспособность кандидата до полной реализации.

| Вариант | Описание | Что даёт | Ресурсы |
|---------|----------|----------|---------|
| **Лабораторный DPI/SNI симулятор** | Стенд с прокси (например, mitmproxy или кастомный middleware), который логирует SNI, JA3/JA4, размеры пакетов и эмулирует блокировку по правилам (блок по SNI, по отпечатку, RST по ключевым словам). Тестовый клиент (скрипт или минимальное приложение) подключается через кандидат-сценарий. | Подтверждение, что трафик не попадает под выбранные правила блокировки; сравнение «до» (текущий XRay/WireGuard) и «после» (новый сценарий). | 1–2 нед на стенд; переиспользование существующего `backend/scripts/vpn_tester.py` и логики проверки конфигов. |
| **Сбор и анализ трафика (proof-of-concept)** | Запись трафика (tcpdump/pcap) при работе прототипа сценария (например, Boring HTTPS или мимикрия) и сравнение с эталонным трафиком целевого сервиса (тот же JA3, размеры, тайминги). | Объективные метрики: насколько трафик похож на эталон; выявление утечек (уникальные заголовки, размеры). | Несколько дней на PoC клиент + скрипты анализа (например, ja3/ja4, размеры первых N пакетов). |
| **Тесты доступности в целевых регионах** | Список эталонных доменов и проверка: резолв через DoH/DoT, доступность по HTTPS с разными SNI, задержки. Можно автоматизировать через периодические запуски (CI или воркер) с гео-нод или партнёрскими точками. | Понимание, какие домены/резолверы реально доступны в регионе; выбор кандидатов для Bootstrap и для «прикрытия» в сценариях 1/4/5. | Зависит от наличия точек в регионах; минимум — ручные проверки по отчётам пользователей или тестовой группе. |

## 2. Тестирование в процессе реализации

| Уровень | Что тестировать | Где в проекте |
|---------|-----------------|----------------|
| **Unit** | Генерация конфига нового туннеля (формат, обязательные поля); логика выбора стратегии (сценарий 3) по результатам зондов; парсинг bootstrap-ответов. | Новые модули в `backend/tests/` по аналогии с `backend/tests/test_xray_manager.py`, `backend/tests/test_connection_orchestrator.py`; в mobile-app — тесты парсера конфига и выбора режима. |
| **Интеграция (backend)** | Создание «клиента» нового типа на сервере, выдача конфига через API, отзыв/отключение. | Расширение `backend/application/services/connection_orchestrator.py` и `backend/application/services/vpn_operations_service.py`; тесты в `backend/tests/` с моками SSH/менеджера для нового протокола. |
| **Интеграция (mobile)** | Получение конфига с бэкенда, передача в новый адаптер туннеля, установка/разрыв соединения, повторное подключение. | `mobile-app/test/integration/vpn_connection_flow_test.dart`, `mobile-app/test/services/vpn_service_*.dart`; добавление сценария «подключение по новому протоколу» и отключение. |
| **E2E (опционально)** | Полный цикл: авторизация → выбор сервера/протокола (в т.ч. новый) → подключение → проверка маршрута (например, запрос к внешнему API с проверкой IP). | Скрипт или отдельный job (реальное устройство/эмулятор с доступом к бэкенду и тестовому серверу); опора на `backend/scripts/vpn_tester.py`. |

## 3. Тестирование после выката

| Вариант | Описание |
|---------|----------|
| **Канареечное развертывание** | Включение нового сценария только для части пользователей (по feature flag или по стране/тегу) с логированием успешных подключений и ошибок. |
| **Сравнение с текущими протоколами** | Метрики: доля успешных подключений, время до первого байта, обрывы по странам/сетям — новый сценарий vs WireGuard/XRay в тех же регионах. |

## 4. Интеграция в сервис

### 4.1. Backend

| Вариант | Описание | Плюсы | Минусы |
|---------|----------|--------|--------|
| **Отдельный менеджер протокола (рекомендуется)** | Новый класс по аналогии с `XrayManager` / `WireGuardManager`: создание/удаление «клиента» на стороне сервера (если у сценария есть серверная сущность), генерация конфига для клиента. Подключение в `ConnectionOrchestrator` и в `VPNOperationsService`: при выборе нового протокола вызывается новый менеджер, конфиг возвращается в том же API (например, `get_config` с полем `protocol` и телом конфига). | Чёткое разделение; переиспользование логики выбора сервера, устройства, подписок. | Нужна реализация менеджера и при необходимости настройка серверов (например, Boring HTTPS endpoint). |
| **Расширение XrayManager** | Если новый сценарий по сути «ещё один транспорт» поверх той же инфраструктуры (например, тот же Xray с другим outbound), добавить вариант в существующий менеджер и в перечисление протоколов. | Меньше новых сущностей. | Подходит только если сценарий укладывается в текущий стек; риск раздувания XrayManager. |
| **Bootstrap как отдельный сервис** | Сценарий 6: отдельный HTTP/HTTPS endpoint (или несколько), отдающий конфиг/список точек входа. Доступ по «прикрывающим» доменам или путям; приложение обращается к нему первым, затем использует полученный конфиг. | Единая точка обновления списка точек входа; можно не светить основной домен приложения. | Нужна инфраструктура (домены, CDN или облако) и обновление мобильного клиента. |

**Реализовано:** Backend: `GET /api/vpn/bootstrap` (без авторизации) возвращает `api_base_urls`, `ttl_seconds` и **`xray_expected_ports`** (`xray_vless`, `xray_vmess`, `xray_reality` — значения совпадают с `backend/core/constants.py`). Запасные URL задаются в бэкенде через `API_BASE_URL_FALLBACKS` (через запятую). Приложение: при старте `AuthService` в фоне вызывает `fetchBootstrapUrls()` и записывает список в `AppConfig.cachedApiBaseUrls`, а порты — в `AppConfig.cachedXrayExpectedPorts` (валидация кэша конфигов Xray через `AppConfig.expectedXrayPort`). Единый список `AppConfig.apiBaseUrls` используется везде. `ApiClient.get/post/put/delete/patch` при таймауте или сетевой ошибке перебирают все URL из `AppConfig.apiBaseUrls`. Запросы VPN (connect, config, disconnect, devices, servers и т.д.) идут через `ApiClient`, поэтому тоже используют перебор баз (bootstrap).

**Ключевые файлы для изменений:**

- `backend/application/services/connection_orchestrator.py` — выбор протокола и сервера, вызов нужного менеджера.
- `backend/application/services/vpn_operations_service.py` — создание/удаление клиента, выдача конфига (единый метод `get_vpn_config(protocol=...)` или расширение существующего).
- Модели сервера/устройства: при необходимости новые поля (например, `boring_https_endpoint`, `mimicry_target`).
- API для мобильного приложения: текущий endpoint выдачи конфига расширить полем `protocol` и форматом конфига под новый сценарий без ломания WireGuard/XRay.

### 4.2. Мобильное приложение

| Вариант | Описание | Плюсы | Минусы |
|---------|----------|--------|--------|
| **Новый адаптер туннеля (рекомендуется)** | По аналогии с `WireGuardAdapter` и `XrayAdapter`: класс с общим интерфейсом (подключение по конфигу, отключение, статус). В `GraniVpnService` при определении протокола из конфига выбирается соответствующий адаптер (например, `BoringHttpsAdapter` или `MimicryAdapter`). | Единый UX «подключился/отключился»; изоляция нативного кода туннеля. | Нужна реализация туннеля на платформе (HTTPS-клиент + TUN или системный VPN API с кастомным сокетом). |
| **Гибридный режим** | Новый сценарий как «транспорт» для существующего стека (например, туннель поверх Boring HTTPS до выхода, дальше тот же XRay). Реализация в одном из существующих адаптеров или тонкий слой поверх. | Быстрее дать выгоду (устойчивый bootstrap) без полной замены протокола. | Сложнее отладка; возможны двойные накладные расходы. |

**Ключевые файлы:**

- `mobile-app/android/app/src/main/kotlin/com/granivpn/mobile/VpnService.kt` — определение протокола (`detectProtocol`), выбор адаптера, вызов `startVpn` с конфигом.
- Новый адаптер (Kotlin/Android) и при необходимости плагин/канал во Flutter для статуса и ошибок.
- Парсинг конфига: расширить `XrayConfigParser` или отдельный парсер под формат нового сценария.

### 4.3. Админ-панель и конфигурация

- **Серверы:** при новом сценарии с собственными endpoint’ами (Boring HTTPS, мимикрия-сервер) — возможность указать URL/домен и параметры (или флаг «включён протокол X» по аналогии с Reality).
- **Протоколы в UI:** новый вариант в списке протоколов только в тех регионах, где он включён (или скрыть под «авто» в сценарии 3).
- **Метрики:** в дашборде при необходимости — разбивка по новому протоколу (подключения, ошибки, средняя длительность сессии).

### 4.4. Поэтапный rollout

1. **Feature flag** в бэкенде и/или в app config: новый сценарий доступен только при включённом флаге (по тегу пользователя или по стране).
2. **Страновой/региональный показ:** включать в первую очередь в регионах с жёсткой блокировкой (по отчёту и метрикам), затем расширять.
3. **Позиционирование в сторах:** описание приложения без формулировок «обход блокировок»; при необходимости гео-ограничение чувствительных функций для соответствия правилам магазинов.
4. **Bootstrap (сценарий 6):** выкат в первую очередь как общее улучшение устойчивости (получение конфига через DoH/прикрывающие домены), затем комбинация с выбранным туннельным сценарием.

## 5. Порядок действий перед выбором реализации

1. Провести **лабораторную оценку** одного–двух приоритетных сценариев (например, 6 + 4 или 6 + 1): PoC трафика и по возможности симулятор DPI/SNI.
2. Зафиксировать **результаты оценки** (документ или раздел в `docs/`): какие сценарии признаны реализуемыми и приоритетными.
3. Определить **вариант интеграции** (отдельный менеджер + новый адаптер vs расширение существующих) и точки изменений в backend/mobile/admin.
4. Реализовать в порядке: **Bootstrap** (если выбран) → **туннель** (4 или 1) → **тесты** (unit → интеграция → при возможности E2E и канарейка).
5. Выкат с feature flag и метриками; сбор обратной связи по регионам и при необходимости A/B против текущих протоколов.
