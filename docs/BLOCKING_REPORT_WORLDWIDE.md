# Углублённый отчёт: блокировки сайтов и сервисов приложений в мире

Документ описывает известные варианты блокировок доступа к сайтам и сервисам приложений, а также технические механизмы, с помощью которых достигается блокировка в каждом случае. Основан на RFC 9505 (IRTF PEARG), исследованиях Censored Planet, Citizen Lab, OONI и открытых источниках.

Связанные документы: [VPN_BYPASS_SCENARIOS.md](VPN_BYPASS_SCENARIOS.md) (новые сценарии обхода), [VPN_BYPASS_TESTING_AND_INTEGRATION.md](VPN_BYPASS_TESTING_AND_INTEGRATION.md) (тестирование и интеграция).

---

## 1. Терминология и три элемента цензуры

Любая техническая блокировка складывается из трёх элементов:

- **Предписание (prescription)** — решение, *что* блокировать (списки доменов, ключевые слова, протоколы, хеши контента).
- **Идентификация (identification)** — *как* в трафике опознать целевой контент или соединение (по заголовкам, SNI, IP, отпечаткам протокола).
- **Вмешательство (interference)** — *как* прервать или ухудшить доступ (сброс соединения, отбрасывание пакетов, подмена DNS, троттлинг).

Ниже описаны конкретные варианты блокировок и способы их достижения.

---

## 2. Точки контроля (где внедряется блокировка)

Блокировка может внедряться в разных точках сети:

| Точка контроля | Кто контролирует | Что можно блокировать |
|----------------|------------------|------------------------|
| **Магистраль / международные шлюзы** | Государство, операторы | Весь входящий/исходящий трафик региона |
| **Провайдеры (ISP)** | Регулятор, по закону | Трафик всех абонентов по доменам/IP/протоколам |
| **Точки обмена трафиком (IXP)** | Операторы, регулятор | Трафик между сетями |
| **CDN и хостинг** | Компании под давлением закона | Контент по URL, региону, контенту |
| **Удостоверяющие центры (CA)** | Государство (выпуск своих сертификатов) | TLS-инспекция, подмена сертификатов |
| **Сервисы и платформы** | Сами компании по закону/политике | Контент, аккаунты, API, приложения |
| **Устройство пользователя** | Предустановленное ПО, корпоративные политики | Любой контент на устройстве |

Дальше — по слоям и типам блокировок.

## 3. Блокировка на уровне приложений (Application Layer)

### 3.1. DNS-блокировка и подмена DNS

**Суть:** Пользователь не может получить «правильный» IP для домена.

**Как достигается:**

- **Рекурсивные резолверы (у ISP или национальных DNS):**  
  На запрос к заблокированному домену возвращают:
  - **NXDOMAIN** («домен не существует»);
  - **подставной IP** (например, страница «доступ ограничен» или 127.0.0.1);
  - **редирект** на другой домен (информационная страница).
- **Блокировка на уровне авторитативных серверов:**  
  Власть или регистратор приостанавливают домен или меняют записи у регистратора/хостинга DNS.

**Примеры:** Китай (GFW), Иран, Россия (резолверы провайдеров), Индонезия, Пакистан, Турция. Китай нередко подставляет «реальные» IP крупных сервисов (Facebook, Twitter), чтобы редирект выглядел как настоящий сайт.

**Ограничения:** Обходится сменой DNS (DoH/DoT, сторонние резолверы), VPN, прямым вводом IP (если известен).

---

### 3.2. Идентификация по HTTP-заголовкам (Host, URL, метод)

**Суть:** Блокировка по полям HTTP-запроса: Host, путь, метод (GET/POST).

**Как достигается:**

- На магистрали или у ISP стоит DPI/прокси.
- Анализируются заголовки HTTP (в незашифрованном трафике).
- При совпадении с блок-листом (домен, путь, ключевые слова в URL) соединение сбрасывается или ответ подменяется.

**Примеры:** Бангладеш, Бахрейн, Китай, Индия, Иран, Малайзия, Пакистан, Россия, Саудовская Аравия, Южная Корея, Таиланд, Турция. Часто используется коммерческое ПО (например, Netsweeper, Sandvine).

**Ограничения:** Не работает для HTTPS без расшифровки; для HTTPS нужна опора на SNI или сертификат (см. ниже).

---

### 3.3. Идентификация по TLS: SNI (Server Name Indication)

**Суть:** В TLS ClientHello клиент передаёт имя хоста (SNI) в открытом виде. По этому имени можно блокировать домен, не расшифровывая тело трафика.

**Как достигается:**

- DPI на пути трафика просматривает ClientHello.
- Извлекается поле SNI (строка с доменом).
- Если домен в блок-листе — пакеты отбрасываются или в канал внедряются TCP RST (обрыв соединения).

**Примеры:** Китай, Египет, Иран, Катар, Южная Корея, Турция, Туркменистан, ОАЭ, Россия (в т.ч. для QUIC с 2022 г.).

**Ограничения:** ESNI/ECH скрывают SNI; в ответ цензоры блокируют все соединения с ESNI (Китай с 2020 г.) или соединения без SNI (Россия).

---

### 3.4. Идентификация по TLS: сертификат сервера

**Суть:** После ClientHello сервер присылает сертификат, в котором указаны домены. По ним можно принять решение о блокировке.

**Как достигается:**

- DPI отслеживает TLS-рукопожатие и извлекает CN/Subject Alternative Names из сертификата.
- Домен из сертификата сверяется с блок-листом; при совпадении — обрыв или дроп.

**Примеры:** Reliance Jio (Индия) и другие операторы. В TLS 1.3 сертификат может быть в зашифрованной части — метод теряет эффективность.

---

### 3.5. Блокировка ESNI и «нестандартного» TLS

**Суть:** Чтобы помешать скрытию SNI, блокируются целые классы соединений.

**Как достигается:**

- По отпечатку TLS-рукопожатия определяется использование ESNI/ECH или отсутствие SNI.
- Все такие соединения дропаются или режутся RST, без разбора по домену.

**Примеры:** Китай (блокировка ESNI с 2020 г.), Россия (блокировка соединений без SNI).

**Ограничения:** Высокий риск over-blocking (страдают легитимные сервисы).

---

### 3.6. Давление на распространителей контента (CDN, хостинг, соцсети)

**Суть:** Блокировка достигается не сетевыми средствами, а через политику платформ и закон.

**Как достигается:**

- Законы об удалении контента (notice-and-takedown, «закон о фейках», лицензии).
- Требования к ключевым словам, хешам изображений/видео, блок-листам.
- Требование отключить сервис или приложение в стране; удаление приложений из локальных каталогов; отключение API для региона.

**Примеры:** Китай (поисковики, соцсети с локальными блок-листами), ЕС (авторские права, контент), Великобритания (блокировка запросов в поиске), Иран/Россия (требования к мессенджерам и соцсетям).

## 4. Транспортный и сетевой уровень (Transport / Network)

### 4.1. TCP RST Injection (внедрение сброса соединения)

**Суть:** Цензор посреди пути отправляет поддельные TCP-пакеты с флагом RST от имени клиента или сервера, чтобы обе стороны считали соединение разорванным.

**Как достигается:**

- Устройство на пути (например, DPI) копирует трафик (port mirroring).
- При совпадении с правилами (ключевые слова, SNI, домен в HTTP) генерируются поддельные TCP RST с подходящими номерами последовательностей и отправляются клиенту и/или серверу.

**Примеры:** Великий файрвол Китая (в т.ч. по ключевым словам в TCP-потоке), Иран, Индонезия (TKPPSE).

**Ограничения:** Если клиент и сервер игнорируют «чужие» RST и проверяют данные, соединение может сохраниться; на практике это сложно для обычных приложений.

---

### 4.2. Отбрасывание пакетов (Packet Dropping)

**Суть:** Пакеты, относящиеся к запрещённому соединению или протоколу, просто не пропускаются.

**Как достигается:**

- По правилам (IP, порт, отпечаток протокола, SNI после DPI) пакеты помечаются как подлежащие блокировке и отбрасываются на маршрутизаторе/фаерволе/DPI.

**Примеры:** Повсеместно в сочетании с другими методами (Россия ТСПУ, Иран, Китай).

---

### 4.3. Блокировка по IP-адресу

**Суть:** Весь трафик на определённый IP (или подсеть) блокируется на сетевом уровне.

**Как достигается:**

- На маршрутизаторах/фаерволе настраиваются ACL: пакеты с destination (или source) из списка IP дропаются или не маршрутизируются.
- Списки обновляются по реестрам регулятора (например, Роскомнадзор в РФ).

**Примеры:** Россия (реестр запрещённых IP), Иран, Китай, многие страны при блокировке хостинга целиком.

**Ограничения:** Over-blocking (на одном IP много сайтов); обход сменой IP (CDN, перенос); усложнение при использовании CDN с общими IP.

---

### 4.4. Идентификация и блокировка по протоколу (DPI)

**Суть:** По отпечатку протокола (размер пакетов, порядок, заголовки) трафик распознаётся как VPN, Tor, кастомный прокси и т.п. и блокируется.

**Как достигается:**

- DPI анализирует первые пакеты соединения и/или статистику потока.
- Сопоставление с известными сигнатурами (OpenVPN, WireGuard, Tor, Shadowsocks и т.д.).
- При совпадении — дроп или RST.

**Примеры:** Китай (блокировка «полностью зашифрованного» трафика по эвристикам с 2021 г.; QUIC по доменам с 2024 г.), Иран (разрешены по сути только DNS, HTTP, HTTPS; блокируются WireGuard, OpenVPN, SSTP, Tor), Россия (развитие ТСПУ в сторону блокировки VPN).

**Ограничения:** Обфускация протокола, маскировка под обычный HTTPS, нестандартные порты усложняют определение; возможен over-blocking.

---

### 4.5. Троттлинг (намеренное замедление)

**Суть:** Доступ не запрещается полностью, но делается настолько медленным, что сервисом пользоваться практически невозможно.

**Как достигается:**

- На DPI/маршрутизаторе трафик к выбранным доменам или приложениям помечается и пропускается с искусственной задержкой или ограничением полосы (rate limiting, shaping).

**Примеры:** Россия — замедление YouTube (2024), ранее Twitter; возможны аналоги в других странах.

---

### 4.6. Манипуляции маршрутизацией (BGP)

**Суть:** Трафик к определённым префиксам «уводится» в никуда или в сеть цензора.

**Как достигается:**

- Объявление более специфичных или «привлекательных» BGP-маршрутов к IP заблокированных ресурсов так, что трафик уходит к провайдеру, который его дропает или фильтрует.

**Примеры:** Описаны в исследованиях; используется не везде из-за сложности и рисков для стабильности интернета.

---

## 5. Блокировка приложений и сервисов (не только сеть)

### 5.1. Удаление из магазинов приложений (App Store / Google Play)

**Суть:** Приложение нельзя официально установить в данной стране/регионе.

**Как достигается:**

- **По решению разработчика:** при публикации выбирается список стран распространения.
- **По требованию властей:** магазин получает предписание и удаляет приложение в юрисдикции или глобально.
- **Политика магазина:** например, приложения, рекламирующие обход «государственных файрволов» или геоблокировок, могут отклоняться (Apple Guideline 5.4); в части стран требуют гео-ограничения внутри приложения (по геолокации), а не только скрытие в сторе.

**Примеры:** Китай (отдельные каталоги приложений, множество западных приложений недоступны), Иран/Россия (удаление по запросам властей), региональные ограничения разработчиков (банки, VPN, стриминг).

---

### 5.2. Геоблокировка на уровне сервиса (backend)

**Суть:** Сервер по IP/геолокации/профилю пользователя не выдаёт контент или отключает функциональность.

**Как достигается:**

- Определение страны по IP (GeoIP), по заголовкам (если прокси передаёт страну), по настройкам аккаунта.
- В ответ для «заблокированной» страны: 403, редирект, пустая выдача, отключение API или критичных функций приложения.

**Примеры:** Стриминг (Netflix, Spotify), соцсети и мессенджеры (разные версии и доступ по странам), банковские и платёжные приложения.

---

### 5.3. Блокировка API и доменов мобильных приложений

**Суть:** Приложение установлено, но не может связаться с сервером.

**Как достигается:**

- Те же сетевые методы: блокировка доменов и IP бэкенда (DNS, IP, SNI, RST) на стороне провайдера или на магистрали.
- Дополнительно: давление на владельца сервиса с требованием отключить API для страны (де-факто блокировка «со стороны приложения»).

**Примеры:** Иран (Telegram 2018 — блокировка доменов и IP Telegram), Россия (блокировка доступа к Twitter, Facebook, мессенджерам через ТСПУ по SNI/доменам).

---

### 5.4. Замедление (троттлинг) для приложений

**Суть:** Как в п. 4.5, но целевой трафик определяется по приложению (например, по доменам/SNI видеосервиса).

**Как достигается:** Трафик к доменам приложения помечается и шейпится/задерживается на уровне ISP/DPI.

**Примеры:** Россия — замедление YouTube, что влияет и на мобильное приложение.

---

## 6. Страновые и региональные системы (краткая сводка)

| Регион/страна | Основные механизмы | Особенности |
|---------------|--------------------|-------------|
| **Китай** | DPI (TSG), DNS-подмена, SNI, RST, блокировка ESNI, эвристики «полностью зашифрованного» трафика, блокировка QUIC по доменам, давление на платформы | Многоуровневая фильтрация на магистрали и у ISP; блок-листы по доменам (HTTP/HTTPS) очень большие |
| **Россия** | ТСПУ (DPI у ISP), реестр доменов/IP, SNI (в т.ч. QUIC), блокировка без SNI, троттлинг, SORM (наблюдение) | Тысячи устройств ТСПУ; развитие в сторону блокировки VPN; замедление YouTube/Twitter |
| **Иран** | DPI на международных стыках, блокировка по протоколу (разрешены в основном DNS/HTTP/HTTPS), блокировка Telegram, VPN, Tor | Жёсткая блокировка протоколов; коллатеральный ущерб при блокировке обходных инструментов |
| **Турция, ОАЭ, Саудовская Аравия** | DNS, HTTP/HTTPS-фильтрация, SNI, блокировка категорий контента (в т.ч. LGBTIQ), коммерческие системы фильтрации | Сочетание сетевой фильтрации и давления на платформы |
| **Индия** | DNS, HTTP, блокировка по сертификату (отдельные ISP), локальные блокировки по регионам | Разный уровень строгости у операторов |
| **Пакистан, Бангладеш** | DNS, HTTP-заголовки, коммерческие DPI-решения | Блокировка по доменам и контенту |
| **ЕС / национальные законы** | Блокировка через суды (DNS/IP у ISP), notice-and-takedown, давление на поисковики и платформы | Юридическое предписание + техническая реализация у ISP и платформ |

## 7. Нетехнические и смешанные меры

- **Изъятие доменов (domain seizure):** приостановка домена регистратором по решению суда или регулятора.
- **Серверы и хостинг:** физическое или юридическое отключение серверов в стране.
- **Notice-and-takedown:** владелец платформы удаляет контент под угрозой санкций.
- **Самоцензура:** блок-листы и фильтры внутри платформ (поиск, соцсети) по требованию властей.
- **Наблюдение (SORM и аналоги):** не блокировка как таковая, но обязательная установка оборудования для доступа спецслужб к трафику; может сочетаться с блокировкой на том же оборудовании.

## 8. Сводная таблица: «как достигается блокировка»

| Вариант блокировки | Где реализуется | Как достигается технически |
|--------------------|-----------------|-----------------------------|
| DNS-блокировка | Рекурсивные резолверы (ISP, гос. DNS) | Подмена ответа на запрос: NXDOMAIN, подставной IP, редирект |
| Блокировка по домену (HTTP) | DPI/прокси у ISP или на магистрали | Анализ Host/URL в открытом HTTP; дроп или RST при совпадении с блок-листом |
| Блокировка по SNI (HTTPS/QUIC) | DPI на пути трафика | Парсинг ClientHello, извлечение SNI; дроп/RST по домену |
| Блокировка по сертификату | DPI | Анализ сертификата в TLS-рукопожатии; в TLS 1.3 менее применимо |
| Блокировка ESNI/без SNI | DPI | Определение типа ClientHello; дроп всех таких соединений |
| TCP RST Injection | Устройство с копией трафика (DPI) | Генерация поддельных TCP RST по правилам (ключевые слова, SNI, домен) |
| Packet dropping | Маршрутизаторы, фаерволы, DPI | Правила по IP/порту/протоколу/SNI; отбрасывание пакетов |
| IP-блокировка | Маршрутизаторы, фаерволы | ACL по списку IP; дроп или отсутствие маршрута |
| Блокировка протоколов (VPN, Tor) | DPI | Отпечатки протокола; дроп или RST при совпадении |
| Троттлинг | DPI + шейпер/политики QoS | Классификация трафика по домену/приложению; задержка или ограничение полосы |
| Удаление из сторов | Магазины приложений | Политика магазина + предписания властей; выбор стран разработчиком |
| Геоблокировка в приложении | Серверы сервиса | GeoIP, настройки аккаунта; отказ в доступе или урезанный функционал |
| Блокировка API/бэкенда приложения | Сеть (DNS, IP, SNI) + возможно давление на владельца | Те же механизмы, что для сайтов; плюс отключение API для страны со стороны сервиса |
| Давление на CDN/платформы | Юридическое + политика платформы | Блок-листы, хеши контента, отключение сервиса в стране |

- **OONI** (ooni.org) — тесты доступности сайтов и приложений по странам.
- **Citizen Lab** — отчёты по цензуре и наблюдению (Иран, Китай, Ближний Восток и др.).
- **Usenix Security / GFW.report** — исследования Великого файрвола Китая (SNI, QUIC, эвристики шифрованного трафика).
- **Freedom House, Freedom on the Net** — обзоры по странам.
- **Internet Society** — документы по DNS-блокировке и политике.

Документ подготовлен в информационных целях для понимания технической стороны блокировок и не призывает к обходу действующего законодательства в какой-либо юрисдикции.
