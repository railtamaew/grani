# Отчёт по оптимизации VPN-сервиса

**Дата:** 2026-02-10  
**Контекст:** Анализ логов, повторное подключение Xray, состояние UI при отключении, сравнение с лучшими практиками.

---

## 1. Повторное подключение (Xray)

**Проблема:** После первого отключения Xray в `disconnect()` обнулялись `_clientId` и `_vpnConfig`. Второе подключение к тому же серверу всегда шло через долгий путь `create-client` (15–30 с), а не через быстрый GET `/vpn/xray/config/$_clientId` (~0.5–2 с).

**Реализованное решение:**
- Введён in-memory кэш `_xrayClientIdCache` по ключу `serverId_protocol`.
- При отключении Xray перед обнулением `_clientId` значение сохраняется в кэш для текущего сервера и протокола.
- При следующем `_connectXray()` для того же сервера/протокола `_clientId` восстанавливается из кэша, срабатывает быстрый путь GET config.
- При ответе 404 или ошибке GET конфига запись из кэша удаляется и выполняется fallback на `create-client`.

**Рекомендация:** Если бэкенд удаляет Xray-клиента при disconnect, текущая схема (кэш + fallback на create-client при 404) остаётся корректной. При необходимости можно дополнительно кэшировать конфиг локально с TTL для офлайн-переподключения.

**Почему повторное подключение всё ещё может быть долгим:** В логах видно: `client_id восстановлен из кэша для 1_xray_reality` → затем `GET config 404, клиент удалён на сервере, create-client`. То есть приложение корректно пробует быстрый путь по кэшированному `client_id`, но бэкенд возвращает 404 — клиент уже удалён на сервере (при disconnect или по таймауту). В этом случае fallback на `create-client` неизбежен, и повторное подключение занимает 15–30 с. Чтобы повторное подключение стало быстрым, нужно, чтобы бэкенд **не удалял** Xray-клиента при disconnect (или сохранял его на время TTL), тогда GET config будет возвращать 200 и быстрый путь сработает.

**Изменения на бэкенде (мгновенное переподключение):** В `ConnectionOrchestrator.disconnect_device()` для протокола Xray реализовано: (1) клиент **не удаляется** с Xray-сервера при отключении (`remove_client` не вызывается); (2) у устройства **не очищаются** `vpn_client_id`, `current_server_id`, `vpn_protocol` — только `is_active = False` и `is_vpn_enabled = False`. В результате при повторном подключении приложение получает GET config 200 по тому же `client_id`, конфиг отдаётся без вызова create-client, переподключение происходит мгновенно.

## 2. Таймауты

| Операция | Текущее значение | Замечание |
|----------|------------------|-----------|
| Отключение через API (`/vpn/disconnect`) | 4 с | В логах возможен `TimeoutException after 0:00:04.000000`. |
| Create-client (первый запрос) | 30 с | Один раз при первом подключении. |
| Create-client (повтор/fallback) | 15 с | При повторных попытках. |
| GET config (Xray) | 5 с | Быстрый путь при повторном подключении. |

**Реализовано:**
- Отключение через API выполняется в режиме fire-and-forget (`unawaited(remoteDisconnect)`): UI не блокируется при таймауте или медленном ответе.
- Таймауты вынесены в `AppConfig`: `disconnectApiTimeout`, `createClientTimeoutFirst`, `createClientTimeoutRetry`, `xrayGetConfigTimeout` — при необходимости можно менять в одном месте или через конфиг.

## 3. ConnectionLogger

**Наблюдение:** В логах встречается 404 «Устройство не найдено» при отправке логов на `/vpn/logs/send`. В этом случае ConnectionLogger выполняет повторную регистрацию устройства (`_registerDeviceRetry`).

**Реализовано:**
- Синхронизация: VpnService передаёт в ConnectionLogger `setCredentials(token, _deviceId!)` после загрузки токена и device_id при инициализации и при connect.
- Ограничение нагрузки при 404: введены `_max404Retries = 3` и `_404RetryCooldown = 5 мин`. После превышения лимита логи не возвращаются в очередь; после cooldown счётчик сбрасывается. При успешной отправке логов счётчик 404 обнуляется.

## 4. Состояние UI при отключении

**Проблема:** Состояние «Отключение…» на кнопке могло не успевать отрисоваться до начала тяжёлых операций в `disconnect()` (80 ms могло быть недостаточно на части устройств).

**Реализованное решение:**
- После `_isDisconnecting = true` и `notifyListeners()` добавлено ожидание одного фрейма через `SchedulerBinding.instance.addPostFrameCallback` и задержку **150 ms** перед продолжением выполнения `disconnect()`.
- Это даёт UI время отрисовать состояние «Отключение…» на разных устройствах (в т.ч. при высокой нагрузке) перед вызовом API и нативного отключения.

**Почему «Отключение» могло не отображаться:** На части устройств одного фрейма и 50 ms было недостаточно для перерисовки. Задержка увеличена до 150 ms после фрейма; при необходимости можно поднять до 200 ms в `vpn_service.dart`.

**Состояния кнопки:** Поддерживаются четыре состояния: off, connecting, disconnecting, on. Обработчик при состоянии ON вызывает `vpnService.disconnect()` (TrialUnifiedScreen, HomeScreen → ConnectionBlock / ButtonConnection).

## 5. Сравнение с лучшими практиками (NordVPN, ExpressVPN, Surfshark и др.)

| Практика | Описание | Статус в приложении |
|----------|----------|----------------------|
| **Быстрое переподключение** | Кэш конфига / «последний сервер», один тап для reconnect. | Реализовано: кэш Xray `client_id`, быстрый путь GET config при повторном подключении к тому же серверу. |
| **Устойчивость** | Авто-reconnect при обрыве сети, опциональный Kill Switch. | Частично: мониторинг соединения и восстановление при обрыве есть; Kill Switch — по необходимости. |
| **Обратная связь** | Явные состояния connecting / disconnecting; уведомления об ошибках без блокировки UI. | Реализовано: четыре состояния кнопки, гарантированное отображение «Отключение…», ошибки через ErrorBanner/неблокирующие сообщения. |
| **Производительность** | Не блокировать главный поток длинными запросами; таймауты и retry для API. | Частично: таймауты заданы; длинные операции в connect/disconnect выполняются в async; при необходимости вынести тяжёлые вызовы в isolate или background. |

## 6. Рекомендации по доработке (приоритизированный список)

1. **Высокий приоритет**
   - Кэш Xray `client_id` для быстрого повторного подключения — **реализовано**.
   - Гарантированное отображение состояния «Отключение…» — **реализовано**.

2. **Средний приоритет**
   - Синхронизация `device_id` с бэкендом и с ConnectionLogger; ограничение повторных попыток при 404 — **реализовано**.
   - Неблокирующее отключение через API (fire-and-forget) — **реализовано**.
   - Настраиваемые таймауты через AppConfig — **реализовано**.

3. **Низкий приоритет**
   - Опциональный Kill Switch и явная настройка авто-reconnect при обрыве сети.
   - Вынос особо тяжёлых операций (например, длительный create-client) из основного потока при необходимости.

## Связанные файлы

- `mobile-app/lib/config/app_config.dart` — таймауты VPN (disconnect, create-client, GET config).
- `mobile-app/lib/services/vpn_service.dart` — кэш Xray, логика disconnect/connect, задержка для UI, fire-and-forget API.
- `mobile-app/lib/services/connection_logger.dart` — отправка логов, ограничение повторов при 404, повторная регистрация устройства.
- `mobile-app/lib/screens/trial_unified_screen.dart`, `mobile-app/lib/screens/main/home_screen.dart` — состояния кнопки подключения.
- `mobile-app/lib/widgets/connection_block.dart`, `mobile-app/lib/widgets/button_connection.dart` — отображение состояний off / connecting / disconnecting / on.
