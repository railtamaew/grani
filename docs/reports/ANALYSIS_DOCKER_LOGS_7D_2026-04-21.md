## A) Краткое резюме

1. За неделю зафиксировано **~115 уникальных `req_id`** на финальной строке **`[session_prepare]`** (итог запроса); из них **108 с `status=200`**, **7 с `401`**, **11 с `500`**.  
2. Все **11 ответов 500** на `session_prepare` приходятся на **один кластер**: **`user_id=1`**, один и тот же **`device_id=3e54f374-84a3-4f3a-8d3c-64a482d2ac83`**, **2026-04-21 09:53–09:57 UTC**, в логах **`create_client_ms=0`** при **`cache=miss`** — типичный признак падения на **`commit`** после удалённого шага (в трассе — **`UniqueViolation`** на **`ux_devices_device_id_active`**). Это **одна корневая причина**, повторённая попытками/retry в UI.  
3. Узкое место по времени для успешного cold path — **`create_client` на ноде**: по строкам с **`cache=miss`**, **`create_client_ms`**: **p50 ≈ 23 ms** (быстрый путь после прогрева/кэша на ноде?), **p90 ≈ 6525 ms**, **max ≈ 7470 ms**; при **`cache=hit`** значения **`create_client_ms`** в логах **0** (ответ из кэша сессии).  
4. **401 «Неверный токен»**: **49** строк `token verification failed`; чаще всего пути **`/api/vpn/servers`** (14), **`/api/vpn/device/push-token`** (14), **`/api/vpn/session/prepare`** (7) — ожидаемо при истечении JWT, но сериями даёт негатив «вылетела авторизация».  
5. **`/api/vpn/device/register`**: в perf видны **400** с текстом про **`NotNullViolation` / `client_logs.device_id`** (несколько раз **2026-04-14** и позже) — **продуктовый баг данных/ORM**, не сетевой шум.  
6. **`/api/payments/google-play/verify`**: **400** «Не удалось верифицировать покупку в Google Play» — риск сценария «оплатил, подписка не включилась» (нужна связка с Play Console / purchase token).  
7. Строки с **`UniqueViolation`** / **`NotNullViolation`** встречаются десятки раз (часть — длинные stack trace на одно событие); для отчёта по инцидентам важнее **уникальный сценарий + req_id**, не сырой count дублей stack.  
8. **`[perf]` с `dt_ms ≥ 500`**: чаще всего **`path=/api/vpn/session/prepare`** (38) и **`/api/vpn/status`** (26) — согласуется с тяжёлым prepare и опросом статуса.  
9. Параллельные запросы: **2** строки `session_prepare` с **`lock_wait_ms > 0`** за неделю — блокировки редки.  
10. Распределение **`user_id` в session_prepare**: **`user_id=1`** — 205 строк (включая stage), **`user_id=23`** — 22; остальной трафик prepare в логах слабо представлен — **смещение выборки** (тестовые/основные аккаунты).

## C) Узкие места и ускорение

- **Кэш `session_prepare`**: уже даёт **`cache=hit`** (~46 итоговых строк с hit vs ~80 miss в агрегате по полю `cache=`); расширять TTL/ключ кэша только с учётом смены сервера/протокола/force-reconnect.  
- **SSH / Xray cold path**: **p90 ~6.5 s** — держать **prewarm** кэша конфига на API (у вас в коде упоминался фоновый прогрев), **пул/переиспользование SSH** к ноде, избегать лишних round-trip на каждый prepare.  
- **БД `devices`**: перед `is_active=True` — **снятие активности с других строк с тем же `device_id`** или смена unique-индекса на `(user_id, device_id)` если бизнес это допускает; плюс **`ORDER BY` в `get_by_device_id`**.  
- **`client_logs` + register_device**: убрать каскад, который выставляет **`device_id=NULL`**, или ослабить NOT NULL / бэкфилл при merge fingerprint.  
- **Токены**: proactive **refresh** на клиенте до истечения; уменьшить всплески 401 на `servers`+`prepare` одним батчем после refresh.  
- **Google Play verify**: ретраи с backoff при 5xx Google, отдельное логирование `purchaseToken` hash (не в открытом виде), алерт при доле 400.  
- **Инциденты с нод**: нормализовать дедуп (не только exact string match), исключить `i/o timeout` на закрытии соединения из «критичных».  
- **`/api/vpn/status` >500ms** в perf (26 раз): проверить N+1 запросы, кэш, индексы по `devices`/`connection_logs`.

| Условие | Что видит пользователь |

| Серия **500** на prepare с одним device_id | «Подписка есть, VPN не включается», кнопка «повторить» бесполезна до смены аккаунта/устройства/данных в БД |

| **400** на `google-play/verify` | «Заплатил — не открылось» |
| Долгий **cold prepare** (6–7 с) | Долгий «Подключение…» без прогресс-индикатора по стадиям |
| Много **xray_log_error** в админке | Паника у поддержки при отсутствии корреляции с успешными сессиями |
