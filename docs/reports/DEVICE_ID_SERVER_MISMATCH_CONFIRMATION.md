# Подтверждение: несовпадение правил device_id в проекте и на VPN-сервере

## Утверждение

**Проблема связана с тем, что device_id в проекте упростили, а на сервере подключение по-прежнему идёт по старым правилам.** Ниже — подтверждение по коду и логам.

---

## 1. Как на сервере определяется клиент (старое правило)

На VPN-сервере в конфиге XRay клиент однозначно задаётся:

- **Email в конфиге:** `{user_id}_{device.id}@granivpn.com`
- **UUID** в этом клиенте — по нему XRay проверяет входящее подключение.

То есть на сервере ключ — **числовой `device.id`** (PK в таблице `devices`), зашитый в email, и связанный с ним UUID.

**Код (не менялся):**

```516:517:backend/services/xray_manager.py
            client_email = f"{device.user_id}_{device.id}@granivpn.com"
            client_id = f"vless_{device.id}_{server.id}"
```

Аналогично в `remote_vpn_manager.py` (строки 935, 948), в `infrastructure/external/xray_manager.py` (266, 335, 415): везде **email = user_id + device.id**.

При подключении клиент шлёт **UUID**. XRay ищет в конфиге клиента с таким UUID (и соответствующим email). Если такого клиента нет — в логах: **`invalid request user id`** (VLESS) или **`user do not exist`** (VMESS).

## 2. Что изменили в проекте (упрощение device_id)

### 2.1 Один «канонический» device на один строковый device_id

- **Репозиторий:** при запросе по строковому `device_id` возвращается **одно** устройство — с максимальными `created_at` и `id` (самое «свежее»).

```18:28:backend/repositories/device_repository.py
    def get_by_device_id(self, device_id: str, user_id: int) -> Optional[Device]:
        """Получить устройство по device_id и user_id"""
        # Важно: в базе могут существовать дубликаты с одинаковым device_id/user_id (исторически).
        # Для стабильной работы всегда берем самое "свежее" устройство.
        return self.db.query(Device).filter(
            and_(
                Device.device_id == device_id,
                Device.user_id == user_id
            )
        ).order_by(Device.created_at.desc().nullslast(), Device.id.desc()).first()
```

- **Дедупликация:** скрипт `dedupe_device_ids.py` оставляет **одну** запись на один строковый `device_id` — с максимальными `created_at` и `id`, остальные деактивирует и обнуляет VPN-поля.

В итоге для одного и того же строкового `device_id` «каноническим» может стать **другая** строка в `devices`, с **другим числовым `device.id`**, чем та, для которой когда-то был создан клиент на VPN-сервере.

### 2.2 Resolve по fingerprint

После переустановки/очистки приложение может получить тот же логический пользователь/устройство по `device_fingerprint`. При этом либо возвращается **старая** запись Device (тот же `device.id`), либо создаётся **новая** с **новым** `device.id`. В последнем случае на сервере клиента с email `user_id_{новый_id}@granivpn.com` ещё нет.

### 2.3 Уникальность активного device_id

Миграция `add_unique_active_device_id.sql` вводит уникальный индекс по `device_id` среди активных устройств. Это закрепляет модель «один активный device_id — одна запись», что согласуется с выбором одного «канонического» устройства и возможной сменой того, какой именно `device.id` считается текущим.

## 3. Цепочка при подключении

1. Приложение шлёт **строковый** `device_id` (UUID приложения) в `POST /api/vpn/connect`.
2. Бэкенд вызывает `validate_and_get_device(user, device_id)` → **`device_repo.get_by_device_id(device_id, user.id)`** → возвращается **одно** устройство с текущими правилами (самое свежее по `created_at`/`id` или оставшееся после дедупа).
3. Для этого устройства берётся **числовой** `device.id` (например, 102).
4. При создании/получении конфига XRay вызывается `create_vless_client(device, server)` (или vmess/reality), где **email = `user_id_102@granivpn.com`** и генерируется/подставляется UUID для этого клиента.
5. Конфиг записывается на сервер **только если** до него доходит успешный вызов `_update_xray_config` и перезапуск XRay. Иначе на сервере остаются только старые клиенты (например, для `device.id` 100 или 101).

Если же приложение использует **закэшированный** конфиг, выданный раньше для **другого** `device.id` (100), то оно подключается с UUID, привязанным к `user_id_100@granivpn.com`. На сервере после упрощения мог остаться только клиент для 102 (или вообще не создаться новый) — тогда XRay не находит пользователя по пришедшему UUID и пишет **invalid request user id** / **user do not exist**.

---

## 4. Подтверждение по логам сервера

В **XRay access.log** на 45.12.132.94 за 11.02.2026:

- Массовые отказы: **`rejected  proxy/vless/encoding: invalid request user id`**, **`invalid request version`**.
- Для VMESS: **`proxy/vmess/encoding: invalid user > user do not exist`**.

Это значит: **UUID (и/или версия запроса), которые шлёт клиент, не соответствуют ни одному клиенту в текущем конфиге на сервере.** То есть либо:

- на сервере остались только клиенты, созданные по **старым** `device.id` (до упрощения/дедупа), а приложение уже использует конфиг для **нового** `device.id`, или  
- на сервере успели записать только **новых** клиентов (новые `device.id`), а приложение ещё подключается со **старым** конфигом (старый UUID),

то есть правила определения «кто такой клиент» в проекте (какой `device.id` считать текущим) и на сервере (какой email/UUID в конфиге) разъехались.

---

## 5. Вывод

- **В проекте** после упрощения device_id используется **одно каноническое** устройство на строковый `device_id` (по `created_at`/`id` или после дедупа), возможно с **новым** числовым `device.id`.
- **На сервере** по-прежнему действует **старое правило**: клиент задаётся парой **(email = `user_id_{device.id}@granivpn.com`, UUID)**. Подключение принимается только если пришедший UUID совпадает с одним из таких клиентов.
- Если текущий в проекте `device.id` не совпадает с тем, для которого в конфиге на сервере реально создан клиент (или конфиг на сервере не обновлялся после смены «канонического» устройства), получаем **invalid request user id** / **user do not exist** и отсутствие подключения по XRay.

Таким образом, утверждение **«проблема связана с тем, что device_id в проекте упростили, а на сервере подключение по старым правилам»** подтверждается и кодом, и логикой выбора устройства, и логами XRay на сервере.
