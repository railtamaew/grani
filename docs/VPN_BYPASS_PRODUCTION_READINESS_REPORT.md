## 3. Условия для работы на боевых серверах

### 3.1 Boring HTTPS на боевом окружении

Чтобы протокол **boring_https** реально работал у пользователей:

1. **URL шлюза туннеля**  
   На бэкенде (или на целевом сервере, например Венгрия) должен быть доступен endpoint туннеля. Варианты:
   - **Встроенный gateway:** тот же хост, что и API. Задать в env:
     ```bash
     BORING_HTTPS_GATEWAY_URL=https://api.granilink.com/api/vpn/tunnel
     ```
   - **Отдельный хост:** поднять сервис, принимающий POST на путь туннеля (тот же формат: 4 байта длина + payload), и задать его URL в `BORING_HTTPS_GATEWAY_URL`.

2. **Доступность с клиентов**  
   URL из `BORING_HTTPS_GATEWAY_URL` должен быть доступен по HTTPS с мобильных устройств (порты 443/80, без блокировок на стороне сети).

3. **SNI (опционально)**  
   Для мимикрии под «прикрывающий» домен можно задать `BORING_HTTPS_SNI` (например, домен CDN). Если не задан, используется хост из `BORING_HTTPS_GATEWAY_URL`.

4. **Список протоколов**  
   При непустом `BORING_HTTPS_GATEWAY_URL` бэкенд уже добавляет `boring_https` в `supported_protocols` для всех серверов в `GET /api/vpn/servers`. Дополнительная настройка серверов в БД для Boring HTTPS не требуется.

**Подтверждение:** Цепочка «бэкенд → конфиг → приложение → POST /tunnel» проверена на локальном бэкенде; формат конфига и ответа gateway совпадает со спецификацией и с тем, что ожидает мобильное приложение.

---

### 3.2 Bootstrap на боевых серверах

- Endpoint `GET /api/vpn/bootstrap` публичный, развёрнут вместе с API.
- Для устойчивости при блокировках можно задать запасные URL в настройках:
  ```bash
  API_BASE_URL_FALLBACKS=https://alt1.example.com/api,https://alt2.example.com/api
  ```
- Приложение уже умеет запрашивать bootstrap и перебирать базы при ошибках.

**Подтверждение:** Формат ответа проверен тестом и лабораторным PoC; приложение использует тот же формат.

---

### 3.3 Сервер Венгрия (один боевой сервер)

Сейчас один сервер (Венгрия) с XRay и WireGuard. Для Boring HTTPS возможно два варианта:

- **Вариант A:** Gateway на том же хосте, что и основной API (например, `https://api.granilink.com/api/vpn/tunnel`). Тогда Венгрия не обязательна для Boring HTTPS — клиенты ходят на API/gateway.
- **Вариант B:** Развернуть gateway на Венгрии (отдельный сервис или прокси к API). Тогда в `BORING_HTTPS_GATEWAY_URL` указать URL этого gateway (например, `https://vpn-hungary.granilink.com/api/vpn/tunnel`). В модели сервера можно ввести поле `boring_https_gateway_url` и подставлять его в конфиг для конкретного сервера; сейчас используется глобальный `settings.boring_https_gateway_url`, и протокол показывается всем серверам.

**Подтверждение:** Логика выдачи конфига и список протоколов не зависят от страны сервера; достаточно одного корректного URL gateway.

## 4. Чек-лист перед включением на бою

- [ ] Задать `BORING_HTTPS_GATEWAY_URL` в окружении бэкенда (URL, доступный с интернета по HTTPS).
- [ ] Убедиться, что endpoint туннеля принимает POST (health и фреймы) и возвращает 200 (проверка скриптом `bypass_lab_tunnel_client.py --url <боевой_url>`).
- [ ] (Опционально) Задать `BORING_HTTPS_SNI` и/или `API_BASE_URL_FALLBACKS` при необходимости мимикрии и устойчивого bootstrap.
- [ ] Убедиться, что мобильное приложение обращается к тому же API (тот же `apiBaseUrl` или полученный через bootstrap), обновить список серверов в приложении и проверить появление протокола Boring HTTPS в селекторе.
- [ ] Один раз провести ручную проверку: выбор сервера → протокол Boring HTTPS → подключение → проверка трафика (например, смена IP или доступ к заблокированному ресурсу).
