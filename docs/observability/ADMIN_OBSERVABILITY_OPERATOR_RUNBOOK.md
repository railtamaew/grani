# Операторский runbook по Admin Observability

Это руководство описывает полный операторский workflow для блока Observability в админке и объясняет все события, которые сейчас генерируются реализацией.

## 1) Для чего нужен этот блок

Страница Observability — единый операционный экран для:

- просмотра нормализованных VPN-событий;
- отслеживания и обработки инцидентов;
- корреляции инцидентов с пользователем/сессией/сервером;
- проверки мониторингового пайплайна тестовыми событиями/инцидентами (только вне production).

Основные разделы UI:

- метрики инцидентов (общее открытых, разрез по severity);
- таблица инцидентов с фильтрами/сортировкой и диалогом деталей;
- таблица событий с фильтрами и cursor-пагинацией;
- генерация тестового события/инцидента.

## 2) Доступ и навигация

Маршрут: `admin-panel` -> `Observability` (пункт в боковом меню).

Ожидаемые роли: `support`, `admin`, `owner`.

## 3) Ежедневный workflow оператора (end-to-end)

### Шаг 1. Триаж инцидентов

1. Откройте `Observability`.
2. Проверьте счетчики: число открытых инцидентов и распределение по severity.
3. В блоке фильтров инцидентов задайте:
   - `Status` (начинайте с `open` и `investigating`);
   - `Severity` (приоритет `P1`, `P2`);
   - `SLA = breached` для on-call режима.
4. Сортируйте по `Last Seen` по убыванию для свежих проблем.

### Шаг 2. Открытие карточки инцидента и оценка масштаба

1. Нажмите `Открыть` в таблице инцидентов.
2. В диалоге:
   - прочитайте `incident_type`, `severity`, `summary`;
   - изучите evidence и превью связанных событий;
   - оцените: изолированная или повторяющаяся проблема.
3. Добавьте комментарий расследования (что проверено, какие гипотезы).

### Шаг 3. Переход из инцидента к потоку событий

1. В диалоге инцидента нажмите `Открыть как фильтр events`.
2. Таблица событий будет предфильтрована по коррелирующим идентификаторам (например, `vpn_session_id`, `server_id`).
3. Навигируйте курсорами:
   - `Следующая` -> более старые страницы;
   - `Предыдущая` -> более новые страницы.

### Шаг 4. Управление жизненным циклом инцидента

Используйте переходы статусов:

- `open` -> инцидент обнаружен, активный анализ еще не начат;
- `investigating` -> активная диагностика;
- `mitigated` -> временная мера применена, продолжаем наблюдать;
- `resolved` -> root cause устранен и подтвержден;
- `ignored` -> принято не обрабатывать (обязателен комментарий).

Перед сохранением задавайте assignee (`Assignee User ID`) и комментарий оператора.

### Шаг 5. Режим live-мониторинга

В разделе событий доступны:

- ручное обновление (`Обновить`);
- `К последним событиям` (переход на самую свежую страницу);
- автообновление каждые 10 секунд без принудительного прыжка из истории;
- уведомление `Новые события доступны` с кнопкой `Показать новые`.

Рекомендация:

- держите автообновление включенным для live-наблюдения;
- при анализе истории используйте уведомление, а не принудительные прыжки.

## 4) Рекомендации по фильтрам

Фильтры событий:

- `User ID`: расследование от пользователя;
- `Server ID`: проблемы на конкретной ноде/сервере;
- `VPN Session ID`: лучший фильтр для одного жизненного цикла туннеля;
- `Event Name`: точечный фокус на конкретном типе событий;
- `Severity`: быстрое сужение по риску.

Фильтры инцидентов:

- `Status`, `Severity`, `Incident Type` для триажа очереди;
- `Server ID`, `Assignee User ID` для маршрутизации ответственности;
- `SLA=breached` для очереди срочного реагирования.

## 5) Severity и ожидания по реакции

Операционный смысл:

- `P1`: критический outage/риск безопасности, немедленная реакция;
- `P2`: серьезная деградация/конфликт, быстрое сдерживание;
- `P3`: умеренная проблема, мониторинг + плановое исправление;
- `P4`: низкое влияние/шум.

Severity событий (`debug|info|warn|error|critical`) — уровень сигнала, а severity инцидентов (`P1..P4`) — уровень реакции.

## 6) Полный каталог событий (как реализовано сейчас)

Важно: ниже перечислены события, которые сейчас реально эмитятся кодом и тестовыми инструментами. Будущая таксономия может быть шире, но этот раздел отражает текущую реализацию.

### A. Backend-события жизненного цикла VPN

- `VPN_CONNECT_PREPARE_STARTED`
  - Значение: начата подготовка сессии/конфигурации перед connect.
  - Типичные поля: `server_id`, `protocol`, `stage=prepare`, `outcome=started`.
  - Примечание: если нет парного `VPN_CONNECT_PREPARE_FINISHED`, проверяйте failure-path запроса.

- `VPN_CONNECT_PREPARE_FINISHED`
  - Значение: подготовка завершилась (`success` или `failure`).
  - Типичные поля: `attrs.status`, `attrs.elapsed_ms`, диагностические поля кэша.
  - Примечание: при ошибках часто есть `reason_code=HTTP_ERROR`.

- `VPN_CONNECT_REQUESTED`
  - Значение: пользователь инициировал подключение VPN.
  - Типичные поля: `user_id`, `server_id`, `protocol`, `stage=connect`.
  - Примечание: стартовая точка для user journey.

- `VPN_TUNNEL_ESTABLISHED`
  - Значение: туннель успешно установлен.
  - Типичные поля: `user_id`, `device_id`, `server_id`, `attrs.ip_address`.
  - Примечание: должен со временем иметь парное `VPN_TUNNEL_CLOSED`.

- `VPN_CLIENT_CONNECTION_ERROR`
  - Значение: connect-флоу завершился исключением.
  - Типичные поля: `reason_code=CONNECT_EXCEPTION`, `message`.
  - Примечание: ключевой error-сигнал нестабильных подключений.

- `VPN_TUNNEL_CLOSED`
  - Значение: disconnect выполнен.
  - Типичные поля: `stage=disconnect`, `attrs.force`.
  - Примечание: используйте вместе с `VPN_TUNNEL_ESTABLISHED` для поиска stale-сессий.

### B. События ingestion клиентских логов

- `VPN_CLIENT_LOGS_RECEIVED`
  - Значение: backend принял payload логов от клиента.
  - Типичные поля: `source=client`, `stage=ingest`, `attrs.logs_count`.
  - Примечание: подтверждает работоспособность telemetry ingestion.

- `VPN_<CLIENT_EVENT_TYPE_UPPER>`
  - Значение: нормализованное событие для каждой строки клиентского лога.
  - Формат источника: `event_name = "VPN_" + upper(event_type)`; fallback `VPN_CLIENT_EVENT`.
  - Маппинг severity: `warn`, если есть `error_code`, иначе `info`.
  - Примеры (зависят от входного payload): `VPN_CONNECTED`, `VPN_RECONNECT`, `VPN_HANDSHAKE_FAILED`.
  - Примечание: каталог здесь намеренно открытый; отслеживайте `event_name` в стриме и поддерживайте словарь для повторяющихся событий.

### C. События конфликтов/целостности для инцидентов

- `VPN_CONFIG_CONFLICT_DETECTED`
  - Значение: обнаружен конфликт/расхождение конфигурации.
  - Используется incident engine для `config_conflict`.

- `VPN_CONFIG_REVISION_MISMATCH`
  - Значение: вариант рассинхронизации revision/hash конфигурации (обрабатывается incident engine).
  - Может приходить из внешних продьюсеров/backfill-пайплайнов.

- `VPN_SESSION_DUPLICATE_DETECTED` (сейчас из testing endpoint)
  - Значение: синтетический сигнал дублирующихся сессий для тестирования/валидации.

- `VPN_SESSION_STALE_DETECTED` (сейчас из testing endpoint)
  - Значение: синтетический сигнал stale-сессии для тестирования/валидации.

## 7) Какие инциденты создаются автоматически

Текущая плановая задача правил `observability.detect_incidents` создает/обновляет:

- `duplicate_session`:
  - триггер: несколько `VPN_TUNNEL_ESTABLISHED` для одного `user_id + device_fingerprint` за 15 минут.
- `stale_tunnel`:
  - триггер: есть `VPN_TUNNEL_ESTABLISHED`, но нет последующего `VPN_TUNNEL_CLOSED` для того же `vpn_session_id`.
- `config_conflict`:
  - триггер: `VPN_CONFIG_CONFLICT_DETECTED` или `VPN_CONFIG_REVISION_MISMATCH`.

Авторазрешение:

- инциденты со статусами open/investigating/mitigated автоматически переводятся в `resolved`, если не обновлялись 30 минут.

## 8) Использование генерации тестовых инцидентов

Кнопка: `Создать тестовый incident/event`.

Поведение:

- создает одно тестовое событие и один открытый инцидент;
- доступно только вне production;
- поддерживает `config_conflict`, `duplicate_session`, `stale_tunnel`.

Используйте только для:

- smoke-проверок UI;
- обучения операторов;
- проверки пайплайна после деплоя.

## 9) Рекомендации по поддержке словаря событий

Когда часто появляются новые динамические клиентские события (`VPN_<...>`):

1. Добавьте/обновите запись словаря через API (`/api/admin/observability/dictionary/events/{event_name}`).
2. Заполните:
   - понятные RU/EN названия;
   - шаблон комментария по умолчанию;
   - операторскую подсказку;
   - severity по умолчанию.
3. Сохраняйте стабильность имен; не переиспользуйте event name для другой семантики.

## 10) Быстрая карта troubleshooting

- Много `VPN_CLIENT_CONNECTION_ERROR` и нет `VPN_TUNNEL_ESTABLISHED`:
  - проверьте auth/config/доступность серверов в connect path.
- Частые циклы `VPN_TUNNEL_ESTABLISHED`/`VPN_TUNNEL_CLOSED`:
  - вероятен reconnect storm или нестабильная сеть.
- Рост инцидентов `config_conflict`:
  - проверьте согласованность rollout (`config_revision`, hash, тайминги apply-очереди).
- События есть, а инциденты не создаются:
  - проверьте Celery beat/задачу `observability.detect_incidents`.
