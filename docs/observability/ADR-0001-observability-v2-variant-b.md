# ADR-0001: Observability V2 (Вариант B)

## Статус

Принято

## Дата

2026-04-27

## Контекст

Текущее логирование в GRANI содержит полезные технические фрагменты, но не формирует надежный и понятный для операторов «единый источник истины» для поддержки и реагирования на инциденты.

Текущие проблемы:

- Сложно проследить серверное состояние VPN сквозным образом (дубли сессий, stale-туннели, конфликты конфигурации).
- Логи фрагментированы между backend, VPN-нодами и клиентской телеметрией.
- `request_id` есть для API-цепочек, но не покрывает полный жизненный цикл VPN-сессии.
- Восстановление пути пользователя требует ручной корреляции и глубокого инженерного контекста.
- В админке есть полезные диагностические endpoints, но нет выделенного incident-centric процесса наблюдаемости.

## Решение

Принять **Вариант B** как целевую архитектуру observability:

- **OpenTelemetry** для инструментирования трасс/логов/метрик в backend и асинхронных воркерах.
- **Loki** для хранения сырых логов и быстрого текстового drill-down.
- **Tempo** для распределенного трейсинга.
- **ClickHouse** для нормализованного event lake и аналитической корреляции.
- **PostgreSQL** для сущностей incident workflow (состояние операторского процесса).
- Модуль «Observability» в админке для таймлайна, инцидентов и user journey.

## Архитектура верхнего уровня

1. Источники
   - FastAPI, Celery-воркеры, внутренние сервисы
   - Инжест логов мобильного клиента
   - Логи VPN-нод (Xray/WireGuard/journalctl)
2. Сбор и маршрутизация
   - OpenTelemetry Collector + Vector/Fluent Bit
3. Хранилища
   - Loki (сырые логи)
   - Tempo (трейсы)
   - ClickHouse (нормализованные события)
   - PostgreSQL (инциденты, словари, действия операторов)
4. Потребители
   - Страницы Observability в админке
   - Дашборды и алерты Grafana
   - Incident engine и уведомления

## Модель корреляции (обязательная)

- `request_id`: идентификатор HTTP/API-цепочки.
- `trace_id`: идентификатор распределенного трейса.
- `vpn_session_id`: идентификатор жизненного цикла VPN (connect -> prepared -> applied -> tunnel_up -> tunnel_down).
- `device_fingerprint`: стабильная идентичность устройства при переустановке/перепривязке.
- `config_revision` / `config_hash_expected` / `config_hash_actual`: идентичность согласованности конфигурации.

`vpn_session_id` — первичный ключ для диагностики поддержки; `request_id` — вторичный.

## Последствия

### Плюсы

- Понятная человеку, incident-oriented диагностика для ops и support.
- Детерминированное обнаружение дублей/stale/конфликтов.
- Снижение MTTR за счет единого таймлайна client/backend/server.
- Масштабируемая аналитика и ретеншн-стратегия через ClickHouse.

### Компромиссы

- Более высокая инфраструктурная сложность и операционные затраты.
- Требуется дисциплина схемы/версионирования для таксономии событий.
- В период миграции нужен dual-write и проверки паритета.

## Что не входит в scope

- Нет прямой замены текущих admin logs endpoints на фазе 1.
- Нет удаления текущих файловых backend-логов до подтверждения паритета.
