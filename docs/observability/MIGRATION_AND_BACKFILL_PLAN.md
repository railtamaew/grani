# План миграции и backfill (Вариант B)

## Цель

Перейти от текущих фрагментированных логов к единому observability-стеку без остановки сервиса.

## Принципы

- Без big-bang переключения.
- Dual-write в переходный период.
- Измеримые пороги паритета между старой и новой диагностикой.
- Готовность операторского UX до перевода на новые экраны по умолчанию.

## План по фазам

## Фаза 0: Подготовка

- Развернуть окружения:
  - OpenTelemetry Collector
  - Loki
  - Tempo
  - ClickHouse
  - Grafana
- Определить SLO по задержке пайплайна и успешности ingestion.

Критерии выхода:

- Инфраструктура доступна и прошла smoke-тесты.

## Фаза 1: Инструментирование + корреляция

- Добавить/прокинуть:
  - `trace_id`
  - `vpn_session_id`
  - существующий `request_id`
- Инструментировать backend и воркеры через OTEL.
- Существующее логирование не менять.

Критерии выхода:

- >=95% connect-флоу содержат `vpn_session_id` и `trace_id`.

## Фаза 2: Нормализация событий

- Реализовать ingestion-transform в каноническую схему.
- Начать запись нормализованных событий в ClickHouse.
- Сохранять сырые логи в Loki.

Критерии выхода:

- Lag событий < 10с (p95).
- Доля ошибок парсинга < 0.5%.

Примечание по backend чтения:

- Контур `/api/admin/observability/events` поддерживает режимы детализации (`basic|enriched`).
- Для тяжелой аналитики должен использоваться внешний backend (`clickhouse|loki`) с сохранением PostgreSQL для incident-workflow и оперативного минимума.
- На текущем шаге включён feature-flag `OBSERVABILITY_EVENTS_BACKEND` с детерминированным fallback до полного включения адаптеров.

## Фаза 3: Incident Engine

- Реализовать правила из `INCIDENT_RULES_SPEC.md`.
- Записывать инциденты в Postgres.
- Включить уведомления (сначала muted/low-priority).

Критерии выхода:

- Дедупликация инцидентов работает, нет неконтролируемого шторма.
- Ручная валидация топ-5 сценариев.

## Фаза 4: Observability в админке

- Выкатить страницы:
  - Live Event Stream
  - User Journey
  - Server Conflicts
  - Incident Center
- Открыть редактор словаря событий для человекочитаемых комментариев.

Критерии выхода:

- Команда поддержки может закрыть тестовый инцидент без CLI.

## Фаза 5: Переключение (cutover)

- Перевести основной troubleshooting на observability-страницы.
- Оставить legacy endpoints read-only на fallback-период.
- Постепенно вывести старые troubleshooting-скрипты.

Критерии выхода:

- 2-4 недели стабильной эксплуатации без критических пробелов.

## Стратегия backfill

Источники:

- `client_logs`, `connection_logs`, `telemetry_events` (Postgres)
- недавние серверные log files / windows из journal

Подход:

1. Построить one-off backfill jobs с детерминированным маппингом событий.
2. Создавать синтетический `vpn_session_id`, если отсутствует:
   - hash от `(user_id, device_id, server_id, time_bucket, protocol)`.
3. Маркировать все backfill-события:
   - `attrs_json.backfill=true`
   - `attrs_json.backfill_batch_id=<id>`

Валидация:

- паритет количества записей по дням между источником и целевым хранилищем
- выборочные проверки воспроизведения таймлайна для известных инцидентов

## План отката

- Если новый пайплайн деградирует:
  - оставить продьюсеры на запись в legacy-логи
  - отключить запись инцидентов rule engine
  - оставить активными legacy diagnostics paths в админке
- Потери данных в основных бизнес-флоу нет, так как observability — side-effect канал.

## Зоны ответственности

- Backend-команда: инструментирование, контракты, нормализация.
- SRE/DevOps: инфраструктура и ретеншн, алерты.
- Команда админки: UI и workflow.
- Лид поддержки: словарь событий и playbook по триажу инцидентов.
