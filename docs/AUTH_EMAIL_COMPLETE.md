# Email Авторизация - Полная документация (GRANI)

**Дата создания:** 2025-01-XX  
**Статус:** ✅ Реализовано и протестировано  
**Версия:** 1.0

## Обзор

Email авторизация в приложении GRANI работает по принципу одноразовых кодов подтверждения. Пользователь вводит email, получает код на почту, вводит код и получает JWT токен для доступа к приложению.

## Поток авторизации

### 1. Ввод Email
- **Экран:** `auth_email_input` (516:206)
- **Действие:** Пользователь вводит email адрес
- **Валидация:** Проверка формата email на клиенте и сервере
- **API:** `POST /api/auth/send-code`

### 2. Отправка кода
- **Экран:** `auth_code_default` (517:387)
- **Действие:** Код отправляется на email пользователя
- **Время действия кода:** 10 минут (`EMAIL_CODE_TTL_MINUTES = 10`)
- **Формат кода:** 4 цифры (например: `1234`)

### 3. Ввод кода
- **Экран:** `auth_code_default` → `auth_register_success` (при успехе)
- **Экран:** `auth_code_default` → `auth_code_wrong` (при ошибке)
- **Действие:** Пользователь вводит 4-значный PIN-код
- **API:** `POST /api/auth/verify-code`
- **Максимум попыток:** 5 (`MAX_VERIFICATION_ATTEMPTS = 5`)

### 4. Успешная авторизация
- **Экран:** `auth_register_success` (517:527)
- **Результат:** Пользователь получает JWT токен
- **Действие:** Переход на экран успешной регистрации

## API Эндпоинты

### POST /api/auth/send-code

**Описание:** Отправляет код подтверждения на email. Создает нового пользователя, если его еще нет.

**Запрос:**
```json
{
  "email": "user@example.com"
}
```

**Ответ при успехе:**
```json
{
  "ok": true,
  "message": "Код отправлен на email",
  "retry_after": null,
  "daily_remaining": 9,
  "daily_sent": 1
}
```

**Ошибки:**
- `400` - Неверный формат email (`invalid_email_format`)
- `429` - Превышен лимит запросов (`rate_limit_exceeded`, `daily_limit_exceeded`)
- `500` - Ошибка отправки email (`email_send_failed`)

**Особенности:**
- Email нормализуется (lowercase, trim)
- Создается новый пользователь, если его нет (`is_verified=False`)
- Код хешируется перед сохранением в БД
- Email отправляется только после успешного сохранения кода в БД

### POST /api/auth/verify-code

**Описание:** Проверяет код подтверждения и выдает JWT токен.

**Запрос:**
```json
{
  "email": "user@example.com",
  "code": "1234"
}
```

**Ответ при успехе:**
```json
{
  "ok": true,
  "token": "jwt_token_here",
  "user": {
    "id": 1,
    "email": "user@example.com",
    "name": "user",
    "is_verified": true,
    "created_at": "2025-01-01T00:00:00",
    "is_blocked": false
  },
  "trialSecondsLeft": 604800,
  "hasActiveSubscription": false
}
```

**Ошибки:**
- `400` - Неверный код (`invalid_code`)
- `410` - Код истек (`code_expired`)
- `429` - Слишком много попыток (`too_many_attempts`)

**Особенности:**
- Код нормализуется (удаляются пробелы)
- Проверяется хеш кода, а не сам код
- После успешной проверки код удаляется из БД
- Пользователь помечается как `is_verified=True`

### POST /api/auth/resend-code

**Описание:** Повторная отправка кода (использует тот же эндпоинт `/send-code`).

**Ограничения:**
- Минимальный интервал между запросами: 30 секунд (настраивается)
- Дневной лимит: настраивается на сервере

## Rate Limiting

### Первый запрос
- Код можно запросить сразу при вводе email
- Нет ограничений на первый запрос

### Повторный запрос
- Минимальный интервал: 30 секунд
- Отображается таймер обратного отсчета в приложении
- Кнопка "Отправить код повторно" активна только когда таймер = 0

### Дневной лимит
- Есть дневной лимит на количество кодов
- При превышении: ошибка `429` с `daily_limit_exceeded`
- Лимит сбрасывается в начале нового дня

## Логика PIN-кода в приложении

### Ввод кода
- Пользователь вводит код по одной цифре
- Автоматический переход между ячейками
- При удалении символа фокус возвращается на предыдущую ячейку
- Проверка кода происходит автоматически при заполнении всех 4 ячеек

### Обработка ошибок
- При неверном коде: показывается экран `auth_code_wrong`
- PIN-блок очищается автоматически
- Можно повторить ввод кода

### Успешная проверка
- Переход на экран `auth_register_success`
- Токен сохраняется в `SecureStorage`
- Информация о пользователе сохраняется в `SharedPreferences`

## Логика таймера повторной отправки

### Начальное состояние
- При входе на экран `auth_code_default` таймер запускается автоматически
- Начальное значение: 30 секунд
- Кнопка "Отправить код повторно" в состоянии `disabled`
- Текст: "Отправить код повторно (30)"

### Работа таймера
- Каждую секунду значение уменьшается: 30 → 29 → 28 ...
- Текст обновляется: "Отправить код повторно (28)", "Отправить код повторно (17)" и т.д.
- При достижении 0:
  - Кнопка становится `active`
  - Текст меняется на: "Отправить код повторно" (без числа)
  - Стиль кнопки меняется согласно макету

### Нажатие активной кнопки
- Отправляется запрос: `POST /api/auth/send-code`
- При успешном ответе:
  - Таймер сбрасывается на 30 секунд
  - Кнопка возвращается в состояние `disabled`
  - Текст: "Отправить код повторно (30)"
- При ошибке:
  - Отображается сообщение об ошибке
  - Таймер не сбрасывается

## Настройка отправки Email

### Провайдер
- **Сервис:** Yandex Cloud Postbox
- **Endpoint:** `https://postbox.cloud.yandex.net`
- **Region:** `ru-central1`
- **Клиент:** `sesv2` (НЕ `ses`)

### Конфигурация
```env
USE_AWS_SES=true
AWS_ACCESS_KEY_ID=your_access_key
AWS_SECRET_ACCESS_KEY=your_secret_key
AWS_SES_ENDPOINT=https://postbox.cloud.yandex.net
AWS_REGION=ru-central1
AWS_SES_FROM_EMAIL=support@granilink.com
AWS_SES_FROM_NAME=GRANI Account
```

### Формат отправителя
- Формат: `"Имя" <email@domain.com>`
- Пример: `"GRANI Account" <support@granilink.com>`

### Шаблон письма
- **HTML шаблон:** Градиентный заголовок от `#032C3C` до `#E69B6E`
- **Заголовок:** "GRANI" (белый, 28px, жирный)
- **Подзаголовок:** "Код подтверждения" (24px)
- **Код:** Большой блок с кодом (36px, letter-spacing 12px)
- **Текст:** "Здравствуйте! Вы запросили код подтверждения для входа в приложение GRANI. Ниже — ваш код:"
- **Время действия:** "Код действителен 10 минут."
- **Предупреждение:** "Если вы не запрашивали этот код — просто проигнорируйте письмо."

### Критические правила
1. **Текстовый ответ = ОШИБКА**
   - Если Yandex Postbox возвращает текстовый ответ - это ошибка
   - Письмо НЕ отправлено
   - НЕ считать текстовый ответ успехом

2. **XML ответ с MessageId = УСПЕХ**
   - Успешная отправка возвращает XML ответ с полем `MessageId`
   - Только в этом случае письмо действительно отправлено

3. **Всегда использовать sesv2 для Yandex Postbox**
   - НЕ использовать старый `ses` client
   - НЕ использовать `SendRawEmail`

## Хранение токена

### Mobile App
- **Токен:** Сохраняется в `FlutterSecureStorage` (ключ: `auth_token`)
- **Email:** Сохраняется в `SharedPreferences` (ключ: `user_email`)
- **User ID:** Сохраняется в `SharedPreferences` (ключ: `user_id`)
- **Auth Type:** Сохраняется в `SharedPreferences` (ключ: `auth_type` = `"email"`)

### Проверка авторизации
- При запуске приложения токен загружается из `SecureStorage`
- Если токен валиден и есть email - пользователь считается авторизованным
- Автоматический переход на экран успешной авторизации при повторном запуске

## UI/UX Правила

### Текстовые поля
- Email вводится без автозамены
- PIN-код вводится по цифрам, автоматический переход между ячейками
- При удалении символа фокус возвращается на предыдущую ячейку PIN
- Ошибки отображаются текстом под полем (НЕ alert-окнами)

### Кнопки
- Primary-кнопки занимают всю ширину контейнера
- Disabled-состояние: уменьшенная прозрачность / отсутствие тени
- Пока выполняется запрос:
  - Кнопка заблокирована
  - Отображается индикатор загрузки
  - Повторные нажатия игнорируются

### Клавиатура
- При открытии системной клавиатуры нижние элементы поднимаются выше середины экрана
- Все поля и кнопки должны оставаться доступными
- Экран содержится в scroll-контейнере

## Безопасность

### Хеширование кода
- Код хешируется перед сохранением в БД
- Используется безопасное хеширование (bcrypt)
- Проверка кода происходит по хешу, а не по самому коду

### Валидация
- Email нормализуется (lowercase, trim)
- Код нормализуется (удаляются пробелы)
- Проверка формата email на клиенте и сервере

### Ограничения
- Максимум попыток ввода кода: 5
- Время действия кода: 10 минут
- Rate limiting на отправку кода
- Дневной лимит на количество кодов
