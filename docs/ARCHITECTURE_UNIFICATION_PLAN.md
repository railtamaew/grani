# План унификации архитектуры проекта

**Дата:** 2026-01-21  
**Версия:** 1.0

## Цель

Создать единую, согласованную архитектуру проекта без противоречий и дублирования.

---

## Этап 1: Backend - Унификация структуры

### Проблема
- Дублирование: `services/` и `application/services/`
- Дублирование: `models/` и `domain/models/`
- Дублирование: `repositories/` и `infrastructure/repositories/`
- Разные импорты в разных файлах

### Решение
**Выбрать структуру:** `domain/application/infrastructure` (Clean Architecture)

**Действия:**
1. Определить, какие файлы используются в API endpoints
2. Удалить неиспользуемые дубликаты
3. Обновить все импорты на единую структуру
4. Удалить старые директории

---

## Этап 2: Mobile - Единый ApiClient

### Проблема
- Каждый сервис создает свой `Dio`
- Разные таймауты
- Нет единых интерцепторов

### Решение
**Создать:** `lib/core/api/api_client.dart`

**Функциональность:**
- Единый `Dio` instance
- Интерцептор для авторизации (добавление токена)
- Интерцептор для обработки ошибок
- Единые таймауты
- Retry логика

---

## Этап 3: Mobile - Logger

### Проблема
- `debugPrint()` везде
- Нет уровней логирования
- Нет структурированного логирования

### Решение
**Создать:** `lib/core/logger/logger.dart`

**Функциональность:**
- Уровни: debug, info, warning, error
- Структурированное логирование
- Возможность сохранения логов
- Замена всех `debugPrint` на `Logger`

---

## Этап 4: Mobile - CacheService

### Проблема
- Прямое использование `SharedPreferences`
- Разные ключи в разных местах
- Нет единого интерфейса

### Решение
**Создать:** `lib/core/cache/cache_service.dart`

**Функциональность:**
- Единый интерфейс для кэширования
- TTL поддержка
- Автоматическая очистка
- Типизированные методы

---

## Этап 5: Mobile - ErrorHandler

### Проблема
- Каждый сервис обрабатывает ошибки по-своему
- Разные форматы извлечения ошибок
- Нет единого подхода

### Решение
**Создать:** `lib/core/errors/error_handler.dart`

**Функциональность:**
- Единый формат ошибок
- Извлечение ошибок из DioException
- Маппинг на пользовательские сообщения
- Централизованная обработка

---

## Этап 6: Mobile - StorageService

### Проблема
- Прямое использование `SharedPreferences` и `FlutterSecureStorage`
- Дублирование логики
- Нет единого интерфейса

### Решение
**Создать:** `lib/core/storage/storage_service.dart`

**Функциональность:**
- Единый интерфейс для хранилища
- Разделение secure и non-secure данных
- Миграция данных
- Типизированные методы
