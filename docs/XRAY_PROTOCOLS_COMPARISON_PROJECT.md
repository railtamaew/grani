# Xray протоколы: сравнение и упрощение в нашем проекте

## Цель документа

- Сравнить `xray_vless`, `xray_reality`, `xray_vmess` в контексте текущей реализации GRANI.
- Зафиксировать, как упростить туннель и уменьшить время подключения.

## Текущая реализация (факт)

- Серверный unit: `xray-v2.service`.
- Активные inbound-порты на HU-BUD-01:
  - `4443` -> VLESS TCP (`security=none`)
  - `2053` -> VLESS REALITY (`security=reality`)
  - `8443` -> VMESS TCP
- На клиенте (mobile-app) уже выбран упрощенный маршрут:
  - основной: `xray_vless`
  - fallback: `xray_reality`

## Сравнение протоколов в нашей эксплуатации

### `xray_vless` (основной)

Плюсы:
- Самый простой путь интеграции и диагностики.
- Предсказуемое поведение в текущем backend/mobile пайплайне.
- Лучше подходит как default для минимального времени коннекта.

Минусы:
- Часть сетей/операторов может резать или деградировать plain TCP-паттерны.

Итог:
- Держим как базовый протокол для большинства подключений.

### `xray_reality` (fallback)

Плюсы:
- Лучше проходит в сетях с ограничениями/фильтрацией.
- Может спасать коннект там, где базовый VLESS деградирует.

Минусы:
- Чувствителен к корректности handshake-параметров.
- В логах чаще встречается шум вида `failed to read client hello` (включая сканеры).
- Усложняет путь, если ставить его в приоритет по умолчанию.

Итог:
- Использовать как fallback, а не как default.

### `xray_vmess` (ограниченный/вспомогательный)

Плюсы:
- Исторически поддерживается системой.

Минусы:
- На публичных портах большой шум `invalid user`.
- Не дает преимущества для текущей цели "быстро и просто".

Итог:
- Не использовать в основном пользовательском маршруте подключения.

## Что реально тормозит туннель (не только протокол)

1. Control-plane шаги до старта data-plane:
   - prepare/create-client/apply-state.
2. Состояние deivce/session веток:
   - лишние ретраи и повторные проверки.
3. Последовательные post-connect probes.
4. Шум на портах (сканеры) и нагрузка на обработку мусорных входящих.

## Упрощенная целевая схема туннеля

1. Проверка валидной session/device связки.
2. Быстрый путь через `GET config` (без SSH).
3. Если нет конфига -> `create-client` (один раз) + apply-state guard.
4. Поднятие `xray_vless` туннеля.
5. Если fail/degraded в окне короткой валидации -> один fallback на `xray_reality`.
6. Если и fallback fail -> controlled error без бесконечных циклов.

## Правила, которые фиксируем

- `xray-v2` — единственный supported unit в operational-контуре.
- `xray_vless` — основной протокол.
- `xray_reality` — только fallback.
- `applied` считается валидным только при:
  - active `xray-v2`,
  - listening inbound портах.
- Любой connect имеет лимит попыток и лимит переключений протокола.

## План в коде (приоритет)

### P0 (срочно)

- Додержать `xray-v2` во всех backend/scripts checks.
- Сохранить fallback-цепочку только `vless -> reality`.
- Включить и поддерживать anti-scan (iptables/fail2ban).

### P1

- Ввести stage-metrics по времени (до `internet_ready`).
- Ускорить post-connect проверки (параллелизация и таймауты).

### P2

- Отдельный "runtime contract" документ для state-machine connect/disconnect.
- Добавить регрессионные тесты на:
  - stale session,
  - быстрый путь `GET config`,
  - fallback ровно один раз.

## Чеклист приемки после изменений

- На Wi-Fi нет типовых сценариев > 60s.
- Порядок протоколов в UI/логике: `xray_vless` выше `xray_reality`.
- Нет ложного `applied` при down-портах.
- Снижение мусорных событий в `xray-v2` логах и стабильнее `connected` сессии.
