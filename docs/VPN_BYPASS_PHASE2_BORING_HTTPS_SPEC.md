## 2.1. Формат конфига для клиента

Конфиг выдаётся бэкендом при выборе протокола `boring_https` (или аналог) и передаётся в мобильное приложение в том же flow, что и конфиги WireGuard/XRay.

### Поля конфига (JSON или структура в API)

| Поле | Обязательное | Тип | Описание |
|------|---------------|-----|----------|
| `protocol` | да | string | `"boring_https"` (или выбранный идентификатор). |
| `endpoint_url` | да | string | Полный URL endpoint туннеля, например `https://gate.example.com/vpn/tunnel`. Определяет хост для TLS (и SNI), порт 443, путь. |
| `sni` | нет | string | Явный SNI; если не задан — берётся хост из `endpoint_url`. |
| `headers` | нет | object | Дополнительные HTTP-заголовки (например, авторизация, кастомный User-Agent). |
| `ttl_seconds` | нет | number | Рекомендуемый TTL сессии/переподключения (опционально). |

### Пример

```json
{
  "protocol": "boring_https",
  "endpoint_url": "https://vpn-gate.example.com/vpn/tunnel",
  "sni": "www.google.com",
  "headers": {
    "Authorization": "Bearer <token>",
    "User-Agent": "Mozilla/5.0 ..."
  }
}
```

Минимальный вариант: только `protocol` и `endpoint_url`.

## 2.2. Протокол «на проводе»

### Установка сессии

1. Клиент открывает TLS-соединение к хосту из `endpoint_url` (или к `sni`, если указан отдельно), порт 443.
2. Целевой домен в SNI — «прикрывающий» (например, известный CDN/поисковик), чтобы трафик выглядел как обычный HTTPS к легитимному сервису.
3. Рекомендуется использовать стек с отпечатком, близким к браузеру (например, BoringSSL на клиенте), чтобы JA3 не выделялся.

### Передача payload туннеля

- **Вариант A (минимальный, как в PoC):** клиент отправляет HTTP POST на путь из `endpoint_url`; в теле — зашифрованный или подписанный payload (трафик туннеля в выбранном формате). Сервер отвечает 200 и в теле отдаёт ответный payload.
- **Вариант B (streaming):** то же, но длинные запросы/ответы с chunked encoding или несколько последовательных POST для фрагментов.
- Формат payload (внутри тела запроса) — на усмотрение реализации: например, длина (4 байта) + данные; или фреймы с типом и телом.

### Обработка на сервере

- Сервер (Boring HTTPS endpoint) принимает TLS, расшифровывает HTTP, по пути/заголовкам определяет, что это туннель.
- Из тела запроса извлекает payload, декодирует, отправляет трафик в интернет (или во внутреннюю сеть), ответ упаковывает и отдаёт в теле HTTP-ответа клиенту.

### Безопасность

- Аутентификация: заголовок (например, `Authorization: Bearer <token>`) или подпись payload. Токен выдаётся бэкендом при создании сессии (create-client / get config).
- Защита от повтора: nonce или счётчик в payload (при необходимости).

## 2.3. Серверная сторона

- **Решение:** отдельный сервис «Boring HTTPS gateway» (не Xray), так как протокол на уровне HTTP, а не VLESS/REALITY.
- **Реализация (минимальная):** endpoint в том же backend — `POST /api/vpn/tunnel` (`api/boring_https_gateway.py`). Health check (пустое тело) → 200 OK. Туннельный формат: 4 байта (big-endian длина) + payload → ответ 200, тело = 4 байта (длина 0) + пустой payload (полная маршрутизация в интернет — TODO).
- Развёртывание: тот же хост, что и API (`boring_https_gateway_url = https://<api-host>/api/vpn/tunnel`), или отдельный хост с проксированием по пути.
- Внутри шлюза: TLS terminator → HTTP → разбор тела → (TODO: маршрутизация трафика в интернет) и обратно.

## 2.4. Точки изменений (напоминание)

| Компонент | Изменения |
|-----------|-----------|
| Backend | Новый менеджер (например, `BoringHttpsManager`): создание «клиента», генерация конфига с `endpoint_url` и токеном; регистрация в ConnectionOrchestrator. |
| Backend API | Ответ get-config: при `protocol=boring_https` отдавать конфиг в формате выше. |
| Mobile (Flutter) | Передача конфига в нативный слой; при необходимости — выбор протокола в UI. |
| Mobile (Android) | Новый адаптер: парсинг конфига, TLS к `endpoint_url` (SNI из конфига), POST с payload, чтение ответа, TUN и маршрутизация. Регистрация в `VpnService`: `when (currentProtocol)` → новый адаптер. |
| Admin | Настройка URL/параметров Boring HTTPS gateway и флаг «протокол включён». |
