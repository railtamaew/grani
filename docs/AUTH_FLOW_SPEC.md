# Документация: Авторизация (GRANI)

## 1. Политика конфиденциальности (Privacy Policy)

### 1.1. Поведение на экранах авторизации
Экран: start_screen, auth_email_input, auth_code_default, auth_code_wrong, auth_code_resend_wait, auth_code_resend_active.
Элемент: текстовая ссылка «Подробнее о конфиденциальности».

**Правила:**
1. Ссылка размещена внизу экрана (как в макете).
2. По клику открывается modal bottom sheet.
3. Bottom sheet отображается поверх текущего экрана, не прерывая авторизационный flow.
4. Bottom sheet закрывается по свайпу вниз или по нажатию на иконку закрытия.
5. Поведение scroll внутри bottom sheet — как в макете.

### 1.2. Поведение в профиле / настройках
Если доступна полноценная страница настроек:
1. Пункт меню «Политика конфиденциальности» открывает отдельный полноэкранный экран.
2. На этом экране контент скроллится вертикально.
3. Навигация: назад через системную стрелку.

### 1.3. Фрейм в макете
Фрейм: bottom_sheet_privacy
Прототип: Open overlay (тип поведения), либо Navigate to — в зависимости от предпочтений, но отображение должно имитировать bottom sheet.

## 2. Компонент: Modal Bottom Sheet / Privacy

Фрейм: bottom_sheet_privacy

**Внутреннее поведение:**
1. Всплывает с нижней части экрана на 60% высоты.
2. Дальнейший drag вверх увеличивает высоту до ~90%.
3. Закрытие:
   - свайп вниз,
   - нажатие на подложку,
   - нажатие на кнопку закрытия (крестик).
4. Контент внутри — вертикальный scroll.
5. Хедер закреплен сверху.
6. Подложка затемнённая, кликабельная.
7. Плавная анимация появления/исчезновения.
8. Удерживает фокус пользователя: при открытии кнопки и поля на нижнем экране неактивны (поведение overlay).
9. При повторном открытии возвращается в начальное состояние (60%).
10. Никакие элементы вне bottom sheet не должны быть доступны для взаимодействия.
11. Bottom sheet всегда центрируется по ширине экрана.
12. Закрытие bottom sheet не должно сбрасывать state родительского экрана.

## 3. Глобальные правила верстки UI

Этот блок применяется ко всем экранам авторизации.
Ссылка в макете: "Документация — Авторизация (для Cursor)".

### 3.1. Клавиатура
1. При открытии системной клавиатуры нижние элементы поднимаются выше середины экрана.
2. Все поля и кнопки должны оставаться доступными.
3. Экран содержится в scroll-контейнере.
4. Верхняя часть экрана не должна резко смещаться.

### 3.2. Текстовые поля
1. При фокусе меняется стиль поля (цвет бордера или тень — строго по макету).
2. Ошибки отображаются текстом под полем.
3. Ошибки не отображаются всплывающими alert-окнами.
4. Email вводится без автозамены.
5. PIN-код вводится по цифрам, автоматический переход между ячейками.
6. При удалении символа фокус возвращается на предыдущую ячейку PIN.

### 3.3. Кнопки
1. Primary-кнопки занимают всю ширину контейнера.
2. Disabled-состояние: по макету (уменьшенная прозрачность / отсутствие тени).
3. Пока выполняется запрос:
   - кнопка заблокирована,
   - отображается индикатор загрузки (по макету),
   - повторные нажатия игнорируются.

### 3.4. Bottom sheet / модалки
1. Initial-высота: примерно 60% экрана, по макету.
2. Расширение drag-жестом до ~90%.
3. Scroll внутри контента.
4. Закрытие свайпом вниз и нажатием на подложку.
5. Фиксированный хедер.
6. Анимация — плавная, по макету.

### 3.5. Навигация и анимации
1. Переходы между экранами авторизации — без анимации (instant).
2. Переходы между основными экранами приложения — dissolve ~150–200 ms.
3. Bottom sheet — slide-up анимация.

## 4. Логика обработки PIN-кода

Экран: auth_code_default → auth_code_wrong → auth_code_resend_* → auth_register_success

**Шаги:**
1. Пользователь вводит PIN-код.
2. При заполнении всех 4 символов приложение отправляет запрос:
   POST /auth/verify-code
3. Если код неверный:
   - показывается экран auth_code_wrong
   - PIN-блок очищается
4. Если код верный:
   - показывается экран auth_register_success

PIN-блок реагирует только после заполнения всех ячеек.

## 5. Документация: Логика таймера повторной отправки кода

Экран: auth_code_default
Элемент: resend_timer_button
Текст: «Отправить код повторно (30)»

### 5.1. Начальное состояние
1. При входе в auth_code_default таймер запускается автоматически.
2. initial_value: 30 секунд.
3. Состояние кнопки: disabled.
4. Отображается текст вида: «Отправить код повторно (30)».

### 5.2. Работа таймера
1. Каждую секунду значение уменьшается: 30 → 29 → 28 ...
2. Текст обновляется соответственно: «Отправить код повторно (28)», «Отправить код повторно (17)» и т.д.
3. При достижении значения 0:
   - кнопка становится active,
   - текст меняется на: «Отправить код повторно»,
   - стиль кнопки меняется согласно макету.

### 5.3. Нажатие активной кнопки
1. Отправляется запрос: POST /auth/resend-code
2. При успешном ответе:
   - таймер сбрасывается снова на 30 секунд,
   - кнопка возвращается в состояние disabled,
   - текст снова отображает «Отправить код повторно (30)».
3. При ошибке:
   - отображается toast-сообщение (как в дизайне),
   - таймер не сбрасывается.

### 5.4. UI/UX требования
1. Пока таймер > 0 кнопка должна быть недоступна.
2. Активное состояние кнопки оформляется согласно макету.
3. Обновление времени происходит раз в секунду.
4. Кнопка не допускает многократных быстрых нажатий.

### 5.5. Структура компонента
Компонент может быть реализован как:
ResendCodeTimer(initialSeconds = 30, onResend = POST /auth/resend-code)
или как логика внутри экрана.

## 6. Переходы в прототипе Figma (для понимания Cursor)

1. start_screen → auth_email_input
2. auth_email_input → auth_code_default
3. auth_code_default → auth_register_success (демонстрация успешного ввода)
4. auth_code_default → auth_code_wrong (ошибка)
5. auth_code_wrong → auth_register_success (повторно введён правильный код)
6. auth_code_resend_active → auth_code_resend_wait (повторная отправка)
7. auth_code_resend_wait → auth_register_success (корректный ввод после resend)

Прототип не отражает backend-логику, это лишь демонстрация путей.
